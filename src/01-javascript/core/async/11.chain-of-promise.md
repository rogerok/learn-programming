# Цепочка промисов

Даже при использовании промисов,
не всегда понятно как структурировать асинхронный код.

Возьмём задачу по объединению двух файлов.

```ts
import fs from "fs";

const unionFiles = (inputPath1, inputPath2, outputPath, cb) => {
  fs.readFile(inputPath1, "utf-8", (error1, data1) => {
    if (error1) {
      cb(error1);
      return;
    }
    fs.readFile(inputPath2, "utf-8", (error2, data2) => {
      if (error2) {
        cb(error2);
        return;
      }
      fs.writeFile(outputPath, `${data1}${data2}`, cb);
    });
  });
};
```

Проведем серию рефакторинга

```js
import fsp from "fs/promises";

const unionFiles = (inputPath1, inputPath2, outputPath) => {
  // Промисы всегда должны возвращаться и строиться в цепочку!
  const result = fsp.readFile(inputPath1, "utf-8").then((data1) => {
    const promise = fsp
      .readFile(inputPath2, "utf-8")
      .then((data2) => fsp.writeFile(outputPath, `${data1}${data2}`));
    return promise;
  });
  return result; // это промис
};
```

Код всё еще структурирован как колбеки, «лесенкой».

```ts
import fsp from "fs/promises";

const unionFiles = (inputPath1, inputPath2, outputPath) => {
  const result = fsp
    .readFile(inputPath1, "utf-8")
    .then((data1) => fsp.readFile(inputPath2, "utf-8"))
    // then ниже берется от промиса readFile
    .then((data2) =>
      // как сюда может попасть data1?
      fsp.writeFile(outputPath, `${data1}${data2}`),
    );
  return result;
};
```

В этой версии есть проблема - если где-то в цепочке ниже нужны данные, которые были получены сверху,
то придется протаскивать сквозь всю цепочку.
Основной выход - создание переменных, через которые данные будут прокинуты дальше:

```ts
import fsp "fs/promises";

const unionFiles = (inputPath1, inputPath2, outputPath) => {
  let data1;

  return fsp.readFile(inputPath1, "utf-8").then((content) => {
    data1 = content;
  }).then(() => fsp.readFile(inputPath2, "utf-8")).then((data2) => fsp.write(outputPath, `${data1}${data2}`))
}
```

Уже не так красиво, но всё еще плоско. Преимущество такого подхода становится всё более и более очевидным с увеличением количества промисов. Тем более далеко не всегда нужно передавать данные дальше.

## Контроль асинхронных операций

Частая ошибка при работе с промисами - потеря контроля.

```ts
import fsp from "fs/promises";

const unionFiles = (inputPath1, inputPath2, outputPath) => {
  const result = fsp.readFile(inputPath1, "utf-8").then((data) => {
    const promise = fsp
      .readFile(inputPath2, "utf-8")
      .then((data2) => fsp.writeFile(outputPath, `${data1}${data2}`));
  });
};
```

Этот код содержит ошибку - константа `primise` не возвращается наружу.
Цепочка промисов прервалась, любая ошибка в цепочке пройдет незамеченной.
Нет гарантии что промис вообще успеет выполниться к тому моменту, когда этого будет ожидать вызывающий код.

**Нужно всегда убеждаться в непрерывности асинхронных операций**

## Динамическая цепочка

Иногда кол-во асинхронных операций заранее неизвестно, но они должны выполняться строго по очереди.
Цепочку можно построить, используя циклы или свертку. Сам принцип останется тот же - ` then().then().then()....`

Единственное что нужно учесть - начальный промис, с которого строится цепочка.
Если такого нет, то можно его создать с помощью ф-ции `Promise.resolve()`
Она возвращает промис, который ничего не делает, но с него можно начать свертку.

```js
const filePaths = /* список путей до файлов */;

// Эта функция принимает на вход необязательное значение,
// которое появится в promise.then((<тут>) => ...)
// Начальное значение в данном случае – массив,
// в котором накапливаются данные из файлов
const initPromise = Promise.resolve([]);

// В then отдается функция, а не ее вызов!
const promise = filePaths.reduce((acc, path) => {
  // Аккумулятор – всегда промис, внутри которого массив с содержимым файлов
  const newAcc = acc.then((contents) =>
    // Читаем файл и добавляем его данные в аккумулятор
    fsp.readFile(path, 'utf-8').then((data) => contents.concat(data)));
  return newAcc;
}, initPromise);

// Если надо, продолжаем обработку
promise.then((contents) => /* обрабатываем все данные полученные из файлов */);
```

## В общем случае в промисах есть два варианта использования `return`

Случай 1 — возврат значения (передача данных в следующий then)
Когда внутри then нужно передать какие-то данные дальше по цепочке, используется return с обычным значением:
Каждый return внутри then передаёт значение в следующий then. Если return не написать — следующий then получит undefined.

```js
fetch("https://api.example.com/user")
  .then((response) => {
    return response.json(); // возвращаем промис — следующий then его "развернёт"
  })
  .then((data) => {
    return data.name; // возвращаем обычное значение
  })
  .then((name) => {
    console.log(name); // получаем "развёрнутое" значение
  });
```

Случай 2 — возврат самого промиса из функции
Если функция содержит промис внутри, но не возвращает его — промис становится неуправляемым снаружи:

```js
const func = () => {
  const promise = new Promise((resolve, reject) => {
    setTimeout(() => {
      console.log("promise done!");
      resolve("result from promise");
    }, 1000);
  });
  // ⚠️ нет return — промис "потерян"
};

func().then((result) => console.log(result)); // ❌ TypeError: Cannot read properties of undefined
```

Чтобы это исправить — нужно вернуть промис из функции:

```js
const func = () => {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      console.log("promise done!");
      resolve("result from promise");
    }, 1000);
  });
};

func().then((result) => {
  console.log(result); // "result from promise"
});
```
