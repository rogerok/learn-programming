Какую проблему решает Traversable на практике?	Traversable помогает переставлять вложенные контейнеры местами, чтобы получить нужный смысл результата: например, превратить Array<Promise<A>> в Promise<Array<A>> или Array<Either<E, A>> в Either<E, Array<A>>.	fp::traversable::problem
Что делает sequence?	sequence переворачивает вложенные контейнеры: из t(f(a)) делает f(t(a)).	fp::traversable::sequence
Что делает traverse?	traverse сначала применяет функцию A -> F<B> ко всем элементам, а затем собирает результат в один общий контейнер F. Это map + sequence.	fp::traversable::traverse
Почему Promise.all(items.map(fn)) связан с traverse?	Потому что это привычный частный случай traverse: сначала map(fn), потом объединение многих Promise в один Promise со списком результатов.	fp::traversable::promise-all
В чём разница между Array<Maybe<A>> и Maybe<Array<A>>?	Array<Maybe<A>> означает, что отдельные элементы могут отсутствовать, а Maybe<Array<A>> означает режим всё-или-ничего для всей коллекции.	fp::traversable::ordering
Когда лучше думать о sequence, а когда о traverse?	sequence нужен, когда у тебя уже есть вложенность F<G<A>>. traverse нужен, когда у тебя есть F<A> и функция A -> G<B>.	fp::traversable::heuristics
Чем players.map(validatePlayer) отличается от traverseEitherArray(players, validatePlayer)?	map возвращает массив Either для каждого игрока отдельно, а traverseEitherArray возвращает один Either для всей коллекции: либо все валидны, либо есть первая ошибка.	fp::traversable::validation
Почему порядок контейнеров меняет поведение программы?	Потому что внешний контейнер задаёт смысл всего вычисления целиком: поэлементную обработку, асинхронность, строгий режим всё-или-ничего или ранний выход по ошибке.	fp::traversable::semantics
Как обычно ведёт себя traverseEitherArray при первой ошибке?	Он останавливает обход и возвращает Left, не продолжая собирать оставшиеся значения.	fp::traversable::either
Почему в TypeScript Traversable обычно выражают через специализированные helper-функции?	Потому что в TypeScript нет встроенных Higher-Kinded Types, поэтому полностью обобщённые сигнатуры вроде F<A> -> G<A> неудобно выражать напрямую.	fp::traversable::typescript
В чём смысл закона идентичности для Traversable?	Нейтральная обёртка не должна менять результат обхода: traverse не должен ломать структуру сам по себе.	fp::traversable::identity-law
В чём смысл закона naturality для Traversable?	Если один контейнер можно честно преобразовать в другой, то делать это до sequence или после sequence должно давать одинаковый результат.	fp::traversable::naturality-law
