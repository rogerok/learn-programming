# Chapter: Контейнеры и `map`

> [!info] Context
> Это упрощённая и более приземлённая версия главы 8 из Mostly Adequate Guide.
> Главная цель главы: понять одну идею, из которой потом вырастают `Maybe`, `Either`, `IO` и `Task`.
>
> **Пререквизиты:** [[../../../core/functions/pure-functions|Чистые функции]], [[../function-composition/function-composition|Композиция функций]]

## Главная идея главы

В обычном коде мы привыкли работать со значением напрямую:

```javascript
const result = value.toUpperCase();
```

В функциональном подходе часто удобнее держать значение внутри специальной обёртки и применять функции **не к самому значению**, а **к значению внутри контейнера**.

Коротко:

- контейнер хранит значение
- `map` применяет функцию к значению внутри контейнера
- результат снова остаётся в контейнере

Это важно, потому что контейнер может добавить полезное поведение:

- остановить вычисление, если значения нет
- сохранить ошибку вместо `throw`
- отложить побочный эффект
- отложить асинхронное вычисление

## Почему эта глава часто кажется сложной

В оригинале быстро сменяются сразу несколько сущностей: `Container`, `Maybe`, `Either`, `IO`, `Task`.

Чтобы не запутаться, полезно всё время держать в голове одну и ту же фразу:

> `map` не вытаскивает значение наружу.  
> `map` говорит контейнеру: "примени функцию к тому, что у тебя внутри".

## 1. `Container`: самая простая коробка

Начнём с самой простой версии. Она почти ничего не умеет, но отлично показывает базовый принцип.

```javascript
class Container {
  constructor(x) {
    this.$value = x;
  }

  static of(x) {
    return new Container(x);
  }

  map(fn) {
    return Container.of(fn(this.$value));
  }
}
```

Пример:

```javascript
const result = Container.of(2)
  .map((x) => x + 3)
  .map((x) => x * 10);

console.log(result); // Container(50)
```

Что здесь происходит:

1. `2` кладётся в контейнер
2. первая `map` применяет `x => x + 3`
3. вторая `map` применяет `x => x * 10`
4. результат остаётся внутри контейнера

Если бы `map` возвращал обычное значение, цепочка после первого шага бы развалилась.

### Зачем нужен такой контейнер

Практической пользы у `Container` почти нет. Он нужен как учебная модель.

Он показывает основу:

- значение можно оборачивать
- функции можно применять через `map`
- вычисление можно продолжать цепочкой

Именно в этом смысле контейнер ведёт себя как **функтор**.

> [!tip] Интуиция
> `Container` можно представить как коробку. Ты не достаёшь предмет из коробки руками, а даёшь коробке инструкцию: "поменяй то, что внутри".

## 2. Что такое функтор без лишней теории

На практическом уровне достаточно такого определения:

> Функтор — это тип, у которого есть `map`, и этот `map` умеет применять функцию к внутреннему значению, не ломая сам контейнер.

То есть:

```javascript
Container.of("hello").map((x) => x.toUpperCase());
// Container("HELLO")
```

Здесь функция работает не "рядом" с контейнером, а **через него**.

## 3. `Maybe`: когда значения может не быть

Проблема из обычного JS:

```javascript
const city = user.address.city.toUpperCase();
```

Если `address` или `city` нет, код падает.

`Maybe` решает эту проблему так: если значения нет, дальнейшие `map` просто ничего не делают.

```javascript
class Maybe {
  constructor(x) {
    this.$value = x;
  }

  static of(x) {
    return new Maybe(x);
  }

  isNothing() {
    return this.$value === null || this.$value === undefined;
  }

  map(fn) {
    return this.isNothing() ? Maybe.of(null) : Maybe.of(fn(this.$value));
  }
}
```

Пример:

```javascript
Maybe.of("hello")
  .map((x) => x.toUpperCase())
  .map((x) => `${x}!`);
// Maybe("HELLO!")

Maybe.of(null)
  .map((x) => x.toUpperCase())
  .map((x) => `${x}!`);
// Maybe(null)
```

### Что важно понять про `Maybe`

- `Maybe` не чинит `null`
- `Maybe` делает отсутствие значения явной частью модели
- если значения нет, цепочка останавливается безопасно

Это очень похоже на встроенную мысль:

```javascript
if (value != null) {
  // тогда продолжай
}
```

Только эта проверка теперь спрятана внутрь контейнера, а не размазана по всему коду.

### Где это полезно

- данные из API
- формы
- поиск первого элемента
- чтение вложенных полей

## 4. `Either`: когда нужно знать причину ошибки

`Maybe` умеет сказать только одно: "значения нет".

Но иногда этого мало. Нужно понимать, **почему** вычисление не удалось.

Для этого используют `Either`:

- `Right` — всё хорошо, внутри полезное значение
- `Left` — произошла ошибка, внутри причина

```javascript
class Left {
  constructor(x) {
    this.$value = x;
  }

  map(_) {
    return this;
  }
}

class Right {
  constructor(x) {
    this.$value = x;
  }

  static of(x) {
    return new Right(x);
  }

  map(fn) {
    return Right.of(fn(this.$value));
  }
}
```

Пример:

```javascript
const parseAge = (value) =>
  Number.isNaN(Number(value))
    ? new Left("Возраст должен быть числом")
    : Right.of(Number(value));

parseAge("42").map((x) => x + 1);
// Right(43)

parseAge("abc").map((x) => x + 1);
// Left("Возраст должен быть числом")
```

### Чем `Either` лучше `throw`

`throw` выкидывает выполнение наружу.  
`Either` оставляет ошибку внутри цепочки как обычное значение специального типа.

Это даёт несколько плюсов:

- ошибка становится частью возвращаемого результата
- код легче композировать
- причина ошибки не теряется

### Разница между `Maybe` и `Either`

| Тип | Что сообщает |
|---|---|
| `Maybe` | "значение есть или его нет" |
| `Either` | "успех или ошибка с объяснением" |

## 5. Почему `map` не достаёт значение наружу

Это центральная мысль всей главы.

Если бы `map` "распаковывал" значение, контейнер бы терял контроль над вычислением.

Но весь смысл контейнера именно в том, что он управляет процессом:

- `Maybe` решает, выполнять функцию или нет
- `Either` решает, продолжать цепочку или оставить ошибку
- `IO` решает, когда произойдёт эффект
- `Task` решает, когда запускать асинхронную работу

Поэтому `map` почти всегда возвращает **тот же тип контейнера**, а не голое значение.

## 6. `IO`: как отложить побочный эффект

Побочные эффекты мешают чистоте функций:

- `console.log`
- чтение из `localStorage`
- работа с DOM
- файловая система
- сеть

Идея `IO`: не выполнять эффект прямо сейчас, а завернуть его в функцию и передать дальше как значение.

```javascript
class IO {
  constructor(fn) {
    this.unsafePerformIO = fn;
  }

  static of(x) {
    return new IO(() => x);
  }

  map(fn) {
    return new IO(() => fn(this.unsafePerformIO()));
  }
}
```

Пример:

```javascript
const io = IO.of("hello")
  .map((x) => x.toUpperCase())
  .map((x) => `${x}!`);

io.unsafePerformIO(); // "HELLO!"
```

### Как это понимать

`IO` не убирает эффект из программы.  
`IO` говорит: "не выполняй его сразу, сначала собери вычисление".

Это полезно, потому что:

- чистую логику можно строить отдельно
- запуск эффекта можно оставить на границу приложения
- код легче тестировать

> [!warning] Важная мысль
> `unsafePerformIO` назван так специально. Это напоминание, что здесь программа наконец-то трогает внешний мир.

## 7. `Task`: асинхронность как контейнер

С асинхронностью проблема похожая: результат появится не сейчас, а потом.

`Task` можно понимать как контейнер для **будущего значения**.

Упрощённый пример:

```javascript
class Task {
  constructor(fork) {
    this.fork = fork;
  }

  static of(x) {
    return new Task((_, resolve) => resolve(x));
  }

  map(fn) {
    return new Task((reject, resolve) =>
      this.fork(reject, (value) => resolve(fn(value)))
    );
  }
}
```

Пример использования:

```javascript
Task.of(2)
  .map((x) => x + 1)
  .map((x) => x * 10)
  .fork(
    (error) => console.error(error),
    (result) => console.log(result)
  );
// 30
```

### Что здесь важно

- `Task` похож на `Promise`, но мыслится как более управляемый и композиционный контейнер
- `map` описывает преобразования будущего результата
- вычисление запускается только в момент `fork`

То есть мы опять видим знакомый паттерн:

- значение пока недоступно напрямую
- мы всё равно можем описывать его преобразование через `map`

## 8. Одна и та же идея во всех контейнерах

| Контейнер | Что лежит внутри | Что особенного делает |
|---|---|---|
| `Container` | обычное значение | просто показывает принцип |
| `Maybe` | значение или пустота | останавливает цепочку при `null`/`undefined` |
| `Either` | успех или ошибка | хранит причину ошибки |
| `IO` | отложенный эффект | не выполняет действие сразу |
| `Task` | будущее значение | откладывает асинхронное вычисление |

## 9. Как это связывается с композицией

Раньше ты композировал обычные функции:

```javascript
const shout = (x) => `${x.toUpperCase()}!`;
```

Теперь появляется следующий уровень:

- функции по-прежнему остаются маленькими и простыми
- но применяются они уже внутри контейнеров через `map`

Это называется **lifting**: обычную функцию можно "поднять" в мир контейнера.

```javascript
const toUpper = (x) => x.toUpperCase();

Maybe.of("hello").map(toUpper);
Either.of?.("hello");
```

Не сама функция становится сложнее.  
Сложнее и полезнее становится контекст, в котором она применяется.

## 10. Краткая шпаргалка

- `Container` — учебная коробка, чтобы увидеть базовую механику
- `Maybe` — используй, когда значения может не быть
- `Either` — используй, когда нужно сохранить причину ошибки
- `IO` — используй, когда хочешь отложить эффект и не смешивать его с чистой логикой
- `Task` — используй, когда работа асинхронная и результат появится позже

## 11. Итог

Если сжать всю главу до трёх предложений:

1. Контейнер хранит значение вместе с некоторым поведением.
2. `map` не вытаскивает значение наружу, а применяет функцию внутри контейнера.
3. За счёт этого можно одинаково описывать обычные вычисления, пустые значения, ошибки, эффекты и асинхронность.

Если эта идея стала понятной, то вся глава 8 уже перестаёт быть набором странных названий и превращается в один повторяющийся паттерн.
