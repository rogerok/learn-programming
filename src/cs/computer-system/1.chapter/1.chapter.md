## 1. Информация — это биты + контекст

### Введение
Вся информация в компьютере — это последовательности битов (0 и 1). Что делает одну последовательность числом, а другую — строкой текста? **Контекст.**

### Обзор
- **Исходная программа** (например `hello.c`) — это текстовый файл, последовательность байтов.
- Каждый байт — это целое число, соответствующее некоторому символу (стандарт ASCII).
- Файлы, содержащие только символы ASCII — **текстовые файлы**. Все остальные — **двоичные файлы**.
- Одна и та же последовательность байтов может представлять: целое число, число с плавающей точкой, строку символов или машинную инструкцию — всё зависит от контекста.

### Углубление
Пример: программа `hello.c` в кодах ASCII:

```
#  i  n  c  l  u  d  e
35 105 110 99 108 117 100 101
```

Символ `#` = байт со значением 35, `i` = 105, и т.д. Символ перевода строки `\n` = 10.

**Ключевой вывод:** машинное представление чисел — это конечные приближения. Они **не тождественны** математическим числам и могут вести себя непредсказуемо (переполнение, потеря точности). Это подробно раскрывается в Главе 2.

### Упражнения
1. Переведите строку `"main"` в коды ASCII вручную (m=109, a=97, i=105, n=110).
2. Почему число `3.14` нельзя точно представить в двоичной системе?
3. Приведите пример, когда одна и та же последовательность байтов интерпретируется по-разному в зависимости от контекста.

| Как называются файлы, состоящие исключительно из символов ASCII? | Текстовые файлы |

## 2. Система компиляции

### Введение
Программа на C — текст, понятный человеку. Машина понимает только двоичные инструкции. Между ними — **система компиляции**, состоящая из четырёх этапов.

### Обзор
Команда `gcc -o hello hello.c` запускает цепочку из четырёх программ:

```
hello.c → [Препроцессор] → hello.i → [Компилятор] → hello.s → [Ассемблер] → hello.o → [Компоновщик] → hello
 (текст)      (cpp)        (текст)      (cc1)        (текст)      (as)       (двоичный)     (ld)      (двоичный)
```

### Углубление

| Этап | Программа | Вход → Выход | Что делает |
|------|-----------|-------------|------------|
| **Препроцессор** | cpp | `.c` → `.i` | Раскрывает директивы `#include`, `#define` — вставляет содержимое заголовочных файлов |
| **Компилятор** | cc1 | `.i` → `.s` | Транслирует C-код в программу на языке **ассемблера** |
| **Ассемблер** | as | `.s` → `.o` | Переводит ассемблер в **машинные инструкции**, упаковывает в перемещаемый объектный файл |
| **Компоновщик** | ld | `.o` + библиотеки → исполняемый файл | Объединяет объектные файлы (например `hello.o` + `printf.o`) в единый исполняемый файл |

Пример ассемблерного кода функции `main`:
```asm
main:
    subq $8, %rsp
    movl $.LC0, %edi
    call puts
    movl $0, %eax
    addq $8, %rsp
    ret
```
Каждая строка — одна низкоуровневая инструкция процессора в текстовой форме.

### Упражнения

3. Объясните, зачем нужен промежуточный этап ассемблера — почему компилятор не генерирует машинный код напрямую?

## 4. Аппаратная организация системы

### Введение
Чтобы понять, что происходит при запуске `./hello`, нужно знать основные компоненты аппаратуры.

### Обзор
Типичная вычислительная система состоит из:

```
┌─────────────────── CPU ───────────────────┐
│  PC  │  Блок регистров  │  ALU            │
└──────────────┬────────────────────────────┘
               │ Системная шина
         ┌─────┴─────┐
         │Мост ввода/ │──── Шина памяти ──── [Память (DRAM)]
         │  вывода    │
         └─────┬─────┘
               │ Шина ввода/вывода
    ┌──────────┼──────────┬──────────────┐
[USB]     [Графика]   [Контроллер    [Сетевой
                       диска]        адаптер]
```

### Углубление

#### Шины
- Электрические проводники, передающие байты между компонентами.
- Передают данные порциями фиксированного размера — **словами**.
- **Размер слова** — фундаментальный параметр системы: 4 байта (32 бита) или 8 байт (64 бита).

#### Устройства ввода/вывода
- Клавиатура, мышь, дисплей, диск — связь с внешним миром.
- Подключены к шине ввода/вывода через **контроллеры** (на материнской плате) или **адаптеры** (платы расширения).

#### Основная память (RAM)
- Временное хранилище программы и данных во время выполнения.
- Физически: микросхемы **DRAM** (Dynamic Random Access Memory).
- Логически: линейный массив байтов, каждый с уникальным адресом (начиная с 0).
- Размер элементов данных зависит от типа: `short` = 2 байта, `int`/`float` = 4 байта, `long`/`double` = 8 байт (на x86_64).

#### Процессор (CPU)
- Механизм, **интерпретирующий** (выполняющий) инструкции из основной памяти.
- **Счётчик команд (PC)** — регистр, хранящий адрес следующей инструкции.
- **Блок регистров** — небольшое количество быстрых ячеек памяти, каждая размером в одно слово.
- **ALU** (Arithmetic/Logic Unit) — вычисляет новые значения данных и адресов.

#### Базовые операции процессора:
| Операция | Описание |
|----------|----------|
| **Загрузка** | Копирует байт/слово из памяти → в регистр |
| **Сохранение** | Копирует байт/слово из регистра → в память |
| **Арифметическая операция** | ALU берёт два регистра, вычисляет результат, кладёт в регистр |
| **Переход** | Извлекает адрес из инструкции → записывает в PC |

**Важное различие:**
- **Архитектурный набор команд (ISA)** — абстрактная модель: какие инструкции есть и что они делают.
- **Микроархитектура** — фактическая реализация: как именно процессор выполняет эти инструкции (конвейер, суперскалярность и т.д.).

### Упражнения
1. Чем контроллер отличается от адаптера?
2. Почему размер слова является фундаментальным параметром системы?

### Карточки Anki

| Вопрос | Ответ |
|--------|-------|
| Что такое шина в контексте аппаратуры? | Совокупность электрических проводников для передачи байтов между компонентами системы |
| Что такое «слово» (word) в контексте процессора? | Порция данных фиксированного размера, передаваемая по шине; обычно 4 байта (32 бит) или 8 байт (64 бит) |

## 5. Выполнение программы hello

### Введение
Проследим весь путь выполнения `./hello` через аппаратуру — от нажатия клавиш до вывода на экран.

### Обзор (три фазы)

**Фаза 1: Ввод команды с клавиатуры**
```
Клавиатура → USB-контроллер → шина → регистр → память
```
Пользователь набирает `./hello`. Каждый символ через контроллер USB передаётся в регистр процессора, а затем сохраняется в основной памяти.

**Фаза 2: Загрузка исполняемого файла**
```
Диск → контроллер диска → (DMA) → основная память
```
После нажатия Enter оболочка загружает файл `hello` с диска в память. Используется **прямой доступ к памяти (DMA)** — данные идут с диска в память, **минуя процессор**.

**Фаза 3: Выполнение и вывод**
```
Память → регистры → ALU → память → графический адаптер → дисплей
```
Процессор выполняет инструкции `main`, копирует байты строки `"hello, world\n"` из памяти через процессор на устройство отображения.

### Углубление
**Ключевое наблюдение:** система тратит огромное количество времени на **перемещение данных** между компонентами. Это копирование — главный враг производительности. Именно поэтому существует кеш-память (следующий раздел).

### Карточки Anki

| Вопрос | Ответ |
|--------|-------|
| Что такое DMA (Direct Memory Access)? | Механизм, позволяющий передавать данные с диска непосредственно в оперативную память, минуя процессор |
| Назовите три фазы выполнения `./hello` | 1) Ввод команды с клавиатуры → память, 2) Загрузка файла с диска → память (DMA), 3) Выполнение инструкций → вывод на дисплей |
| Что является главным «узким местом» при выполнении программы? | Перемещение (копирование) данных между компонентами системы |

## 6. Кеш-память и иерархия памяти

### Введение
Процессор работает в ~100 раз быстрее основной памяти, а память — в ~10 000 000 раз быстрее диска. Кеш-память — мост через эту пропасть.

### Обзор

**Проблема разрыва скоростей:**
- Регистры: единицы наносекунд, сотни байт
- Основная память: десятки наносекунд, гигабайты
- Диск: миллисекунды, терабайты

**Решение — кеш-память (cache):**

| Уровень | Технология | Размер | Скорость | Что кеширует |
|---------|-----------|--------|----------|-------------|
| L0 | Регистры | ~сотни байт | Самые быстрые | — |
| L1 | SRAM | десятки КБ | Почти как регистры | Данные из L2 |
| L2 | SRAM | сотни КБ – МБ | В 5 раз медленнее L1 | Данные из L3 |
| L3 | SRAM | МБ | В 5–10× медленнее L1 | Данные из основной памяти |
| L4 | DRAM (основная память) | ГБ | ~100× медленнее регистров | Данные с диска |
| L5 | Локальные диски | ТБ | ~10 000 000× медленнее памяти | Данные из сети |
| L6 | Удалённые хранилища | — | Самые медленные | — |

### Углубление

**Принцип иерархии памяти:**
Память одного уровня служит **кешем** для следующего нижнего уровня.

**Принцип локальности:**
Программы обычно обращаются к данным и коду, расположенным в **ограниченных областях памяти**. Два вида:
- **Временна́я локальность** — если обратились к адресу, скоро обратятся снова.
- **Пространственная локальность** — если обратились к адресу, скоро обратятся к соседним.

Кеш эксплуатирует локальность: хранит «горячие» данные ближе к процессору.

**Практический вывод:** программист, знающий о кеше, может ускорить программу **на порядки**, правильно организовав доступ к данным (см. Главу 6).

### Упражнения
1. Почему нельзя просто сделать всю память из SRAM?
2. Объясните, почему проход по массиву быстрее, чем случайный доступ к его элементам, с точки зрения кеша.
3. Какой вид локальности использует цикл `for (i=0; i<n; i++) sum += a[i]`?

### Карточки Anki

| Вопрос | Ответ |
|--------|-------|
| Из какой технологии изготовлена кеш-память L1–L3? | SRAM (Static Random Access Memory) |
| Из какой технологии изготовлена основная память? | DRAM (Dynamic Random Access Memory) |
| В чём основная идея иерархии памяти? | Память одного уровня служит кешем для следующего, более медленного уровня |

## 7. Операционная система и её абстракции

### Введение
Программа `hello` не обращается к клавиатуре, диску или памяти напрямую. Между программой и железом стоит **операционная система** — слой, предоставляющий три ключевых абстракции.

### Обзор

```
Прикладные программы
─────────────────────────────
Операционная система
   Процессы │ Виртуальная │ Файлы
            │   память     │
─────────────────────────────────
Процессор │ Основная память │ Устройства I/O
```

**Две задачи ОС:**
1. Защита аппаратуры от катастрофических действий программ
2. Предоставление простого единообразного интерфейса к сложному оборудованию

**Три абстракции ОС:**

| Абстракция | Что скрывает |
|------------|-------------|
| **Файлы** | Устройства ввода/вывода |
| **Виртуальная память** | Основную память + диск |
| **Процессы** | Процессор + память + устройства I/O |

### Углубление

#### 7.1 Процессы
- **Процесс** — абстракция выполняющейся программы. Каждый процесс «думает», что он единственный в системе.
- ОС обеспечивает **иллюзию** параллельного выполнения через **переключение контекста**.
- **Контекст** = значение PC + регистры + содержимое памяти процесса.
- Переключение: ОС сохраняет контекст текущего процесса → восстанавливает контекст нового → передаёт ему управление.
- Переход из приложения в ядро осуществляется через **системный вызов**.
- **Ядро (kernel)** — не отдельный процесс, а блок кода и данных, управляющий всеми процессами. Всегда находится в памяти.

#### 7.2 Потоки (threads)
- Процесс может состоять из нескольких **потоков выполнения**.
- Потоки разделяют код и глобальные данные процесса.
- Потоки эффективнее процессов для параллельных вычислений (меньше накладных расходов).
- Подробно — Глава 12.

#### 7.3 Виртуальная память
Каждый процесс имеет своё **виртуальное адресное пространство**:

```
Старшие адреса (верх)
┌──────────────────────────────┐
│ Виртуальная память ядра       │  ← недоступна приложению
├──────────────────────────────┤
│ Пользовательский стек         │  ← растёт вниз при вызовах функций
├──────────────────────────────┤
│ Разделяемые библиотеки        │  ← printf и др.
├──────────────────────────────┤
│ Динамическая память (куча)    │  ← malloc/free, растёт вверх
├──────────────────────────────┤
│ Данные (чтение/запись)        │  ← глобальные переменные
├──────────────────────────────┤
│ Код и данные (только чтение)  │  ← загружается из исполняемого файла
└──────────────────────────────┘
Младшие адреса (низ, начало программы)
```

| Область | Описание |
|---------|----------|
| Код и данные | Фиксированный размер, загружается из исполняемого файла, адрес начала одинаков для всех процессов |
| Куча (heap) | Динамически растёт/сжимается (`malloc`/`free`) |
| Разделяемые библиотеки | Код и данные общих библиотек (libc и др.) |
| Стек | Реализует вызовы функций, растёт/сжимается автоматически |
| Память ядра | Зарезервирована для ОС, приложение не может читать/писать |

**Основная идея:** содержимое виртуальной памяти хранится на диске, а основная память используется как кеш для часто используемых страниц. Подробно — Глава 9.

#### 7.4 Файлы
- **Файл** — последовательность байтов. Точка.
- Каждое устройство I/O моделируется файлом: диск, клавиатура, дисплей, сеть.
- Все операции I/O — через системные вызовы чтения/записи файлов → **подсистема ввода/вывода Unix**.
- Унифицированный интерфейс: программа работает одинаково на разных дисковых технологиях.

## 8. Обмен данными в сетях

### Введение
Сеть — это ещё одно устройство ввода/вывода. С точки зрения системы, сетевой адаптер — такое же устройство, как диск.

### Обзор
- Система копирует байты из памяти в сетевой адаптер → данные уходят на другую машину.
- Другая машина читает данные из сети → сохраняет в своей памяти.
- Примеры сетевых приложений: email, веб, FTP, telnet.

**Пример: запуск `hello` на удалённой машине через telnet:**
1. Пользователь вводит `"hello"` → клиент отправляет строку серверу
2. Сервер передаёт строку своей командной оболочке
3. Оболочка запускает `hello`, передаёт вывод серверу
4. Сервер пересылает `"hello, world\n"` клиенту
5. Клиент выводит строку на дисплей

## 9. Закон Амдала

### Введение
Закон Амдала — количественный ответ на вопрос: **«На сколько ускорится система, если ускорить одну её часть?»**

### Обзор

Пусть:
- `α` — доля времени, которую занимает ускоряемая часть
- `k` — во сколько раз мы ускорили эту часть

Тогда **общее ускорение**:

```
        1
S = ─────────────
    (1 - α) + α/k
```

### Углубление

**Пример:**
Часть системы занимает 60% времени (α = 0.6). Ускорили её в 3 раза (k = 3).

```
S = 1 / (0.4 + 0.6/3) = 1 / (0.4 + 0.2) = 1 / 0.6 = 1.67×
```

Несмотря на ускорение части в **3 раза**, общее ускорение — лишь **1.67×**.

**Предельный случай (k → ∞):**

```
S∞ = 1 / (1 - α)
```

Если α = 0.6, то даже при бесконечном ускорении этой части: S∞ = 1/0.4 = **2.5×**.

**Главный вывод закона Амдала:**
Чтобы значительно ускорить систему, нужно ускорить **бо́льшую часть** системы, а не маленький фрагмент.

### Упражнения

**Упражнение 1.1 (из книги):**
Грузовик едет 2500 км за 25 часов (100 км/ч). На 1500 км пути отменили ограничение скорости — можно ехать 150 км/ч.
1. Каково общее ускорение рейса?

**Упражнение 1.2 (из книги):**
Маркетинг пообещал ускорение ПО в 2 раза. Вы можете ускорить только 80% системы (α = 0.8). Какое `k` нужно?

**Решение 1.2:**
```
2 = 1 / (0.2 + 0.8/k)
0.2 + 0.8/k = 0.5
0.8/k = 0.3
k = 0.8/0.3 ≈ 2.67
```

| Вопрос | Ответ |
|--------|-------|

| Часть системы занимает 60% времени. Ускорили в 3 раза. Общее ускорение? | 1.67× (по формуле: 1/(0.4 + 0.2) = 1/0.6) |
| Каков предел ускорения при α = 0.6, если k → ∞? | S∞ = 1/(1-0.6) = 2.5× |
| В чём главная идея закона Амдала? | Для значительного ускорения системы нужно ускорять бо́льшую часть системы, а не маленький фрагмент |

## 10. Конкуренция и параллелизм

### Введение
Две движущие силы развития вычислительной техники: делать **больше** (конкуренция) и делать **быстрее** (параллелизм).

### Обзор

| Термин | Определение |
|--------|------------|
| **Конкуренция (concurrency)** | Одновременное выполнение множества действий (может быть иллюзией — через быстрое переключение) |
| **Параллелизм (parallelism)** | Использование конкуренции для реального ускорения работы системы |

Три уровня параллелизма:

### Углубление

#### Уровень 1: Конкуренция на уровне потоков
- Несколько программ/потоков выполняются «одновременно».
- На однопроцессорной системе — иллюзия через переключение контекста.
- **Многоядерные процессоры** — несколько ядер на одном кристалле, каждое со своими L1/L2 кешами, общий L3.
- **Гиперпоточность (Hyperthreading / SMT)** — одно ядро может выполнять несколько потоков, имея дублированные PC и регистры, но общий ALU. Пример: Intel Core i7 — 4 ядра × 2 потока = 8 потоков.

#### Уровень 2: Параллелизм на уровне инструкций
- Процессор выполняет **несколько инструкций за один такт**.
- **Конвейерная обработка (pipelining)** — инструкция разбивается на стадии; стадии разных инструкций работают параллельно.
- **Суперскалярные процессоры** — выполняют >1 инструкции за такт.
- Подробно — Главы 4 и 5.

#### Уровень 3: SIMD (Single-Instruction, Multiple-Data)
- Одна инструкция обрабатывает **несколько элементов данных** параллельно.
- Пример: сложить 8 пар чисел `float` одной инструкцией.
- Основное применение: обработка изображений, звука, видео.
- Подробно — приложение в интернете OPT:SIMD к Главе 5.

### Карточки Anki

| Вопрос | Ответ |
|--------|-------|
| Чем конкуренция отличается от параллелизма? | Конкуренция — одновременное выполнение множества задач (может быть иллюзией). Параллелизм — использование конкуренции для реального ускорения |
| Что такое гиперпоточность (Hyperthreading)? | Технология, при которой одно ядро процессора может выполнять несколько потоков за счёт дублирования некоторых ресурсов (PC, регистры) |
| Что такое конвейерная обработка (pipelining)? | Разбиение выполнения инструкции на стадии, которые работают параллельно для разных инструкций |
| Что такое суперскалярный процессор? | Процессор, способный выполнять более одной инструкции за такт |
| Что такое SIMD? | Single-Instruction, Multiple-Data — одна инструкция параллельно обрабатывает несколько элементов данных (например, 8 float за раз) |
| Назовите три уровня параллелизма от высшего к низшему | 1) Потоки/процессы, 2) Инструкции (конвейер, суперскалярность), 3) Данные (SIMD) |

## 11. Абстракции в компьютерных системах

### Введение
Абстракция — одна из важнейших концепций информатики. Она позволяет использовать код, не вникая в детали реализации.

### Обзор

Четыре ключевых абстракции компьютерной системы:

```
              Виртуальная машина         ← абстрагирует весь компьютер
                    │
    Архитектурный набор инструкций (ISA) ← абстрагирует процессор
                    │
         Виртуальная память              ← абстрагирует память
                    │
              Файлы                      ← абстрагируют устройства I/O
```

| Абстракция | Скрывает |
|------------|---------|
| **Файлы** | Разнообразие устройств ввода/вывода |
| **Виртуальная память** | Физическую организацию RAM + диск |
| **ISA** | Реальную микроархитектуру процессора |
| **Виртуальная машина** | Всю систему (ОС + процессор + программы) |

**API** (Application Program Interface) — пример абстракции на уровне ПО: набор функций, позволяющий использовать код, не зная его внутреннего устройства.

### Карточки Anki

| Вопрос | Ответ |
|--------|-------|
| Назовите четыре уровня абстракции в компьютерной системе (снизу вверх) | Файлы → Виртуальная память → ISA → Виртуальная машина |
| Что такое API в контексте абстракций? | Application Program Interface — набор функций, позволяющий использовать код без знания его внутреннего устройства |
| Какую абстракцию обеспечивает ISA? | Скрывает реальную микроархитектуру процессора за абстрактной моделью набора команд |
