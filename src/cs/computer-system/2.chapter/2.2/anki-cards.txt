Как работает B2U (Bits to Unsigned)? Что значит u8(0b10110100)?	Каждый бит — положительная гиря. Вес = степень двойки по позиции. 0b10110100 = 128 + 32 + 16 + 4 = 180. Просто перевод из двоичной в десятичную.	csapp::ch2.2::b2u
Чем B2T (signed) отличается от B2U (unsigned)?	Одно отличие: старший бит — «отрицательная гиря». Для u4 гири: +8, +4, +2, +1. Для i4 гири: -8, +4, +2, +1. Только знак у старшего бита.	csapp::ch2.2::b2t
Как прочитать i4(0b1011) — число со знаком?	Старший бит = 1, значит отрицательная гиря включена: -8. Остальные: +2 +1 = 3. Итого: -8 + 3 = -5.	csapp::ch2.2::b2t
Что такое T2U — перевод signed в unsigned? Что происходит с битами?	Биты НЕ меняются — меняется только способ чтения. Как открыть один файл в двух программах. Если число отрицательное, unsigned-значение = число + 2^w. Пример: i16(-12345) и u16(53191) — одинаковые биты 0xCFC7.	csapp::ch2.2::t2u
Что такое U2T — перевод unsigned в signed? Когда значение меняется?	Биты те же. Если unsigned-значение > TMax (верхняя половина диапазона), signed-значение = число - 2^w. Пример: u16(53191) → i16(-12345). Числа 0..TMax читаются одинаково.	csapp::ch2.2::u2t
Какие биты у -1 в two's complement? Почему?	Все единицы: 0xFF для i8, 0xFFFFFFFF для i32. Потому что: -8(отрицательная гиря) + 4 + 2 + 1 = -1 для i4. Все положительные гири включены, но не дотягивают до нуля ровно на 1.	csapp::ch2.2::b2t
Четыре особых битовых паттерна: 0, -1, TMin, TMax — какие биты?	0 = все нули [0000]. -1 = все единицы [1111]. TMin = один старший бит [1000] (только отрицательная гиря). TMax = все кроме старшего [0111] (все положительные гири без отрицательной).	csapp::ch2.2::паттерны
Почему -TMin вызывает переполнение в i8?	TMin_8 = -128, TMax_8 = 127. Число 128 не влезает в i8! Отрицательных на одно больше, чем положительных. В Zig: -(@as(i8, -128)) — паника.	csapp::ch2.2::асимметрия
TMin, TMax, UMax для i8/u8, i16/u16, i32/u32 в hex?	i8: TMin=0x80, TMax=0x7F, UMax=0xFF. i16: TMin=0x8000, TMax=0x7FFF, UMax=0xFFFF. i32: TMin=0x80000000, TMax=0x7FFFFFFF, UMax=0xFFFFFFFF. Паттерн: TMin = 8000..., TMax = 7FFF..., UMax = FFFF...	csapp::ch2.2::диапазоны
@bitCast vs @intCast в Zig — в чём разница?	@bitCast — биты остаются, тип меняется (как T2U/U2T). @intCast — проверяет, что значение влезает в новый тип, иначе паника. Пример: @bitCast(i8(-1)) → u8(255). @intCast(i8(-1)) в u8 → ПАНИКА.	csapp::ch2.2::zig
@truncate vs @intCast при сужении типа — когда какой?	@truncate — отрезает старшие биты, никогда не паникует. Используй когда усечение намеренное (младшие 8 бит хеша). @intCast — паника если не влезает. Используй как «проверочный каст». std.math.cast — возвращает null если не влезает.	csapp::ch2.2::zig
Zero extension vs sign extension — как расширять число к более широкому типу?	Unsigned: дописываем нули слева. u8(200)=0xC8 → u16=0x00C8. Signed: копируем знаковый бит влево. i8(-56)=0xC8 → i16=0xFFC8 (единицы слева). Значение в обоих случаях не меняется.	csapp::ch2.2::расширение
Почему -1 < 0U даёт false в C?	C молча превращает -1 в unsigned: биты 0xFFFFFFFF читаются как 4294967295. Сравнение: 4294967295 < 0 = false. В Zig такой код не скомпилируется — нужен явный каст.	csapp::ch2.2::ловушки
Три ловушки unsigned: обратный цикл, length-1 при 0, вычитание длин. Как чинить?	1) Обратный цикл: while (i > 0) { i -= 1; ... } — уменьшаем внутри. 2) Вместо i <= len-1 пиши i < len (при len=0: 0-1 = огромное число!). 3) Вместо a.len - b.len > 0 пиши a.len > b.len. Золотое правило: не вычитай unsigned.	csapp::ch2.2::ловушки
Что происходит при преобразовании i16(-12345) → u32? Два шага.	Шаг 1: sign extension i16→i32 — копируем знаковый бит: 0xCFC7 → 0xFFFFCFC7 (всё ещё -12345). Шаг 2: reinterpret i32→u32 — те же биты, но unsigned: 4294954951. Порядок важен!	csapp::ch2.2::расширение::ловушки
