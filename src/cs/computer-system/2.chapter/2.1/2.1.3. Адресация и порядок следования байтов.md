# 2.1.3. Адресация и порядок следования байтов

**Програмнный объект** - переменная или структура данных, которая занимает память

```C
int x
double y
struct User {...}
```

Если int = 4 байта → значит объект занимает 4 подряд идущих байта в памяти.

Практически во всех машинах такие объекты хранятся в виде непрерывных последовательностей байтов

В одних машинах такие последовательности будут храниться от наименьшего значащего байта к наибольшему (от младшего к старшему, младший = "самый правый") (обратный или остроконечный порядок, little-endian).

```
int x = 0x12345678
12 34 56 78
```

Будет лежать в памяти в таком виде

```
| Адрес | Значение |
| ----- | -------- |
| 0x100 | 78       |
| 0x101 | 56       |
| 0x102 | 34       |
| 0x103 | 12       |

```

В других - наоборот (прямой, тупоконечный порядок, big-endian)

Для большинства прикладных программистов порядок следования байтов полностью невидим.
Программы скомпилированные для любого класса машин, дают идентичные результаты.

**Иногда порядок байтов становится проблемой**

1. Когда двоичные данные передаются по сети между разными машинами.
   Возникает случай, когда данные, созданные машиной с little-endian, отправляются на машину с bid-endian.
   Чтобы этого избежать, отправляющая машина должна гарантированно преобразовать данные в сетевой стандарт
2. Порядок байтов важен, потому что одно и то же число может по-разному храниться в памяти (little-endian vs big-endian), и без учёта этого невозможно корректно интерпретировать последовательность байтов. Ошибка в интерпретации приводит к тому, что значения (например, адреса или числа) читаются неверно, что вызывает баги, сбои программы или неправильную работу с бинарными данными.
