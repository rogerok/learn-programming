Какие числа можно точно представить в двоичной дроби?	Только числа вида x/2^k: 0.5, 0.25, 0.375, 5.75... Число 0.1 = 0.0001100110011...₂ (бесконечная периодическая дробь). Аналогия: можно нарезать пирог на половины, четверти, восьмые — но не на десятые.	cs::csapp::float::binary-fractions
Три поля IEEE 754 float. Сколько бит в f32 и f64?	sign (1 бит) + exponent + fraction. f32: 1+8+23=32. f64: 1+11+52=64. sign — знак числа, exponent — масштаб (окно), fraction — позиция внутри окна.	cs::csapp::float::ieee754
Модель «окно и смещение» (window and offset) для IEEE 754.	Exponent = «окно»: определяет интервал [2^E, 2^(E+1)). Fraction = «смещение»: делит окно на 2^23 (f32) равных частей, указывает позицию. Пример: 3.14 → окно [2,4), смещение 57% от начала.	cs::csapp::float::ieee754
Три категории чисел IEEE 754. Как определить категорию?	1) Нормализованные: exp ≠ 0 и exp ≠ все 1. E = exp-bias, M = 1.frac. 2) Денормализованные: exp = 0. E = 1-bias, M = 0.frac. 3) Специальные: exp = все 1. frac=0 → ±Infinity, frac≠0 → NaN.	cs::csapp::float::categories
Что такое «неявная единица» в нормализованных числах?	Мантисса нормализованного числа всегда имеет вид 1.xxx, поэтому начальную 1 не хранят — она подразумевается. Это даёт бесплатный дополнительный бит точности. M = 1.fraction (а не 0.fraction).	cs::csapp::float::normalized
Зачем нужны денормализованные числа?	Заполняют разрыв между нулём и наименьшим нормализованным числом. Обеспечивают «плавный переход к нулю» (gradual underflow). Без них — скачок от 0 сразу к ~1.18e-38 (для f32). Аналогия: режим «супер-зум» микроскопа.	cs::csapp::float::denormalized
Почему NaN != NaN? Как проверить на NaN?	По стандарту IEEE 754: NaN не равен ничему, включая себя. Проверка: std.math.isNan(x) в Zig, Number.isNaN(x) в JS, или трюк x != x (true только для NaN).	cs::csapp::float::nan
Чем +0.0 отличается от -0.0?	При == они равны: +0.0 == -0.0. Но 1.0/(+0.0) = +Infinity, 1.0/(-0.0) = -Infinity. Минус ноль появляется при -1.0 * 0.0 = -0.0.	cs::csapp::float::zero
Сколько десятичных цифр точности у f32 и f64?	f32: ~7.2 цифры (23 бита мантиссы). f64: ~15.9 цифр (52 бита). «double вдвое точнее float» — неверно: 7 vs 16, это в ~2.2 раза.	cs::csapp::float::precision
При каком числе f32 перестаёт различать соседние целые?	2^24 = 16 777 216. При этом числе шаг между соседними f32 = 1.0. Значит 16777216.0f32 + 1.0 == 16777216.0f32. Для f64 граница: 2^53 = 9 007 199 254 740 992.	cs::csapp::float::integer-limit
Почему float-числа распределены неравномерно на числовой прямой?	В каждом окне [2^E, 2^(E+1)) одинаковое количество чисел (2^23 для f32). Но окна растут экспоненциально: [1,2) шириной 1, [2,4) шириной 2, [1024,2048) шириной 1024. Относительная точность одинакова, абсолютная — нет.	cs::csapp::float::density
Четыре режима округления IEEE 754. Какой по умолчанию?	1) К ближайшему чётному (default). 2) К нулю. 3) Вниз (floor). 4) Вверх (ceil). По умолчанию — round-to-nearest-even: при числе ровно посередине выбирается значение с чётным младшим битом. Аналогия: банковское округление.	cs::csapp::float::rounding
Почему округление «к ближайшему чётному» лучше обычного?	При миллионах операций обычное «вверх при .5» создаёт систематическое смещение вверх. Round-to-even чередует: 1.5→2, 2.5→2, 3.5→4, 4.5→4. В среднем ошибка нулевая.	cs::csapp::float::rounding
Ассоциативно ли сложение float? Пример.	НЕТ. (1e20 + -1e20) + 3.14 = 3.14. Но 1e20 + (-1e20 + 3.14) = 0.0, потому что 3.14 поглощается при добавлении к -1e20. Компилятор НЕ имеет права переставлять float-слагаемые без fast-math.	cs::csapp::float::associativity
Какие алгебраические свойства есть и каких нет у float-арифметики?	ЕСТЬ: коммутативность (+, *), монотонность (кроме NaN). НЕТ: ассоциативность (+, *), дистрибутивность. NaN*0 = NaN (не 0).	cs::csapp::float::properties
Три встроенные функции Zig для преобразования float/int.	@floatFromInt — int→float (округление если не точно). @intFromFloat — float→int (к нулю, паника при NaN/Inf/overflow). @floatCast — между float-типами (f64→f32 теряет точность). Также @bitCast для реинтерпретации бит.	cs::csapp::float::zig
Что такое @setFloatMode в Zig? Два режима.	.strict (default): строгое IEEE 754, компилятор не переставляет операции. .optimized: компилятор может переупорядочивать float-операции (аналог -ffast-math). В .optimized (a+b)+c может стать a+(b+c).	cs::csapp::float::zig
Типы float в Zig. Что такое comptime_float?	f16, f32, f64, f80, f128, comptime_float. comptime_float — произвольная точность при компиляции (не ограничен IEEE 754). При приведении к runtime-типу действуют обычные правила округления.	cs::csapp::float::zig
Что такое «замёрзший одометр» для float?	При накоплении суммы f32 счётчик «замерзает» на 2^24 = 16 777 216: прибавление 1.0 округляется до 0. Решение: f64 (граница 2^53), или алгоритм Kahan summation, или целочисленный счётчик.	cs::csapp::float::pitfalls
Что такое катастрофическая потеря точности (catastrophic cancellation)?	При вычитании двух почти равных чисел значащие цифры взаимно уничтожаются, результат состоит из шума округления. Пример: 1.0000000000000002 - 1.0000000000000001. Относительная ошибка результата огромна.	cs::csapp::float::pitfalls
Почему float опасен как счётчик цикла?	Цикл while (x != 1.0) { x += 0.1; } может быть бесконечным: 0.1 не представимо точно, и после 10 итераций x может не стать ровно 1.0. Решение: целочисленный счётчик.	cs::csapp::float::pitfalls
Почему 0.1 + 0.2 != 0.3?	0.1 и 0.2 — бесконечные двоичные дроби, хранятся с округлением. Сумма округлённых значений = 0.30000000000000004, а 0.3 хранится как 0.29999999999999999. Эти два приближения не совпадают.	cs::csapp::float::pitfalls
Когда сравнение float через == корректно?	Когда число получено точным присвоением (не арифметикой): x == 0.0, x == 1.0. Сравнение с infinity. Проверка x != x для NaN. Не корректно: сравнение результатов вычислений без epsilon.	cs::csapp::float::comparison
