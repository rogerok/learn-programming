## Упражнение 1: Двоичные дроби --- ручной перевод (уровень: легко)

Переведи следующие числа в двоичную систему (до 8 знаков после точки):

1. `6.5`
2. `0.375`
3. `9.1875`
4. `0.1` (покажи, что дробь бесконечная)

> [!hint]- Подсказка 1
> Вспомни, что каждый бит справа от точки — это степень 1/2:
> - первый бит = 0.5
> - второй бит = 0.25
> - третий бит = 0.125
> - четвёртый бит = 0.0625
>
> Для `6.5`: сначала переведи целую часть (6 = 110₂), потом дробную (0.5 = .1₂).

> [!hint]- Подсказка 2
> Для перевода дробной части: умножай на 2, бери целую часть.
> Пример для 0.375:
> - 0.375 × 2 = 0.75 → бит = 0
> - 0.75 × 2 = 1.5 → бит = 1
> - 0.5 × 2 = 1.0 → бит = 1
> Результат: 0.011₂

> [!hint]- Ответы
> 1. `6.5 = 110.1₂` (4 + 2 + 0.5)
> 2. `0.375 = 0.011₂` (0.25 + 0.125)
> 3. `9.1875 = 1001.0011₂` (8 + 1 + 0.125 + 0.0625)
> 4. `0.1 = 0.0001100110011...₂` — бесконечная периодическая дробь, потому что 1/10 не является дробью вида x/2^k

Проверь свои ответы в Zig:

```zig
const std = @import("std");

pub fn main() void {
    std.debug.print("6.5    = {d:.20}\n", .{@as(f64, 6.5)});   // точно: 6.5
    std.debug.print("0.375  = {d:.20}\n", .{@as(f64, 0.375)}); // точно: 0.375
    std.debug.print("9.1875 = {d:.20}\n", .{@as(f64, 9.1875)}); // точно
    std.debug.print("0.1    = {d:.20}\n", .{@as(f64, 0.1)});   // неточно!
}
```

## Упражнение 2: Разбор битов f32 (уровень: легко)

Для числа `-6.5` вручную определи:
1. Знаковый бит (sign)
2. Окно (какой интервал `[2^E, 2^(E+1))`?)
3. Смещение внутри окна (какой процент?)
4. Значение exponent (с bias)
5. Значение fraction

> [!hint]- Подсказка 1
> Знак: число отрицательное, значит sign = 1.
> Для окна — работай с абсолютным значением 6.5. Оно лежит между какими степенями двойки? Между 4 и 8.

> [!hint]- Подсказка 2
> Окно [4, 8): ширина = 4.
> Смещение = (6.5 - 4) / 4 = 2.5 / 4 = 0.625 (62.5% от начала окна).
> Номер окна E = 2 (потому что [4, 8) = [2², 2³)).
> exponent = E + bias = 2 + 127 = 129 = 10000001₂.

> [!hint]- Ответ
> - sign = 1
> - Окно: [4, 8), E = 2
> - Смещение: 62.5%
> - exponent = 129 = 10000001₂
> - fraction = 0.625 × 2²³ = 5242880 = 10100000000000000000000₂
> - Итого: `1 10000001 10100000000000000000000`

Проверь через `@bitCast` в Zig:

```zig
const std = @import("std");

pub fn main() void {
    const val: f32 = -6.5;
    const bits: u32 = @bitCast(val);

    const sign: u1 = @truncate(bits >> 31);
    const exp: u8 = @truncate(bits >> 23);
    const frac: u23 = @truncate(bits);

    std.debug.print("sign: {}, exp: {} (binary: {b:0>8}), frac: 0x{x:0>6}\n", .{
        sign, exp, exp, @as(u32, frac),
    });
}
```

## Упражнение 3: Три категории чисел (уровень: средний)

Определи категорию (нормализованное, денормализованное, специальное) и значение для каждого битового паттерна f32:

1. `0 10000001 10100000000000000000000`
2. `1 00000000 00000000000000000000000`
3. `0 11111111 00000000000000000000000`
4. `0 00000000 10000000000000000000000`
5. `1 11111111 10000000000000000000000`
6. `0 01111111 00000000000000000000000`

Для каждого ответь:
- Какая категория?
- Чему равно число (или какое специальное значение)?

> [!hint]- Подсказка: как определить категорию?
> Смотри на exponent:
> - Если exponent = `00000000` → **денормализованное**
> - Если exponent = `11111111` → **специальное** (Infinity или NaN)
> - Всё остальное → **нормализованное**
>
> Для нормализованных: bias = 127, E = exp - 127, M = 1.fraction
> Для денормализованных: E = 1 - 127 = -126, M = 0.fraction

> [!hint]- Подсказка к вычислению значения (пример для №1)
> Паттерн: `0 10000001 10100000000000000000000`
> - sign = 0 (положительное)
> - exp = 10000001₂ = 129, E = 129 - 127 = 2
> - fraction = 101₂ = 1/2 + 1/8 = 0.625, M = 1.625
> - Значение = +1 × 1.625 × 2² = 6.5

> [!hint]- Ответы
> 1. Нормализованное, +6.5
> 2. Денормализованное, -0.0 (отрицательный ноль)
> 3. Специальное, +Infinity
> 4. Денормализованное, наименьшее положительное с fraction = 0.5 → 0.5 × 2^(-126) ≈ 5.88e-39
> 5. Специальное, NaN (fraction ≠ 0)
> 6. Нормализованное, +1.0 (exp = 127 → E = 0, M = 1.0 → 1.0 × 2⁰ = 1.0)

## Упражнение 4: Граница целочисленной точности (уровень: средний)

Напиши программу на Zig, которая находит **наименьшее** целое число `n`, при котором `@as(f32, @floatFromInt(n)) + 1.0 == @as(f32, @floatFromInt(n))`.

> [!hint]- Подсказка 1
> Вспомни из главы: у f32 мантисса 23 бита + 1 неявный бит = 24 бита. Значит f32 точно представляет целые числа до 2²⁴. Ответ будет рядом с этим числом.

> [!hint]- Подсказка 2
> Не перебирай все числа с 1 — начни поиск с 16_000_000 (рядом с 2²⁴ = 16 777 216). Так программа отработает за секунду.

> [!hint]- Ответ
> `n = 16 777 217` (2²⁴ + 1). При этом числе f32 не может представить два соседних значения 16777216 и 16777217 — оба округляются до 16777216.0. Поэтому + 1.0 ничего не меняет.

```zig
const std = @import("std");

pub fn main() void {
    var n: u32 = 16_000_000;
    while (true) {
        const f: f32 = @floatFromInt(n);
        if (f + 1.0 == f) {
            std.debug.print("Граница f32: {}\n", .{n});
            break;
        }
        n += 1;
    }
    // Бонус: найди аналогичную границу для f64
}
```

## Упражнение 5: Нарушение ассоциативности (уровень: средний)

Найди три числа `a`, `b`, `c` типа `f32`, для которых:
1. `(a + b) + c != a + (b + c)`
2. `(a * b) * c != a * (b * c)`

Объясни, какое именно промежуточное округление вызывает разницу.

> [!hint]- Подсказка для сложения
> Нужны числа с **огромной** разницей в масштабе. Когда маленькое число прибавляется к огромному — оно «поглощается». Попробуй: a = 1e20, b = -1e20, c = 3.14.

> [!hint]- Подсказка для умножения
> Нужны числа, при перемножении которых возникает переполнение или потеря точности. Попробуй: a = 1e20, b = 1e20, c = 1e-20.

> [!hint]- Ответ (сложение)
> `(1e20 + -1e20) + 3.14 = 3.14` — сначала 1e20 и -1e20 взаимно уничтожаются, получаем 0, потом + 3.14.
> `1e20 + (-1e20 + 3.14) = 0.0` — сначала -1e20 + 3.14 = -1e20 (3.14 поглощается), потом 1e20 + (-1e20) = 0.

```zig
const std = @import("std");

pub fn main() void {
    const a: f32 = 1e20;
    const b: f32 = -1e20;
    const c: f32 = 3.14;

    std.debug.print("(a+b)+c = {d:.2}\n", .{(a + b) + c}); // 3.14
    std.debug.print("a+(b+c) = {d:.2}\n", .{a + (b + c)}); // 0.00
    std.debug.print("Равны? {}\n", .{(a + b) + c == a + (b + c)}); // false
}
```

## Упражнение 7: Округление к ближайшему чётному (уровень: средний)

Вручную округли следующие двоичные числа до 2 знаков после точки (оставь 2 бита после точки), используя правило round-to-nearest-even:

1. `10.01010₂` (= 2.3125)
2. `10.01110₂` (= 2.4375)
3. `10.11100₂` (= 2.875)
4. `11.00100₂` (= 3.125)

Для каждого укажи:
- Два ближайших представимых значения (с 2 битами после точки)
- Ровно посередине или нет?
- Результат округления и почему

> [!hint]- Подсказка: что значит «к чётному»?
> «Чётное» в двоичной системе = последний сохраняемый бит равен 0.
> Правило работает только когда число ровно посередине (отбрасываемые биты = 100...0).
> Если не посередине — просто округляй к ближайшему.

> [!hint]- Подсказка к примеру №3
> `10.11100₂`: сохраняем 2 бита после точки, отбрасываем `100`.
> Ближайшие: `10.11₂` (2.75) и `11.00₂` (3.0).
> Разница одинакова → ровно посередине.
> `10.11₂` — младший бит = 1 (нечётное). `11.00₂` — младший бит = 0 (чётное).
> Результат: `11.00₂` = 3.0.

> [!hint]- Ответы
> 1. `10.01010₂`: между `10.01₂` (2.25) и `10.10₂` (2.5). Ближе к 2.25 → **`10.01₂` = 2.25**
> 2. `10.01110₂`: между `10.01₂` (2.25) и `10.10₂` (2.5). Ближе к 2.5 → **`10.10₂` = 2.5**
> 3. `10.11100₂`: ровно посередине между `10.11₂` (2.75) и `11.00₂` (3.0). К чётному → **`11.00₂` = 3.0**
> 4. `11.00100₂`: ровно посередине между `11.00₂` (3.0) и `11.01₂` (3.25). К чётному → **`11.00₂` = 3.0**

## Упражнение 8: «Замёрзший одометр» (уровень: средний-сложный)

Напиши программу, которая суммирует массив f32 двумя способами:
1. **Наивно:** слева направо
2. **С компенсацией ошибки:** алгоритм Кэхэна (Kahan summation)

Сравни результаты для массива из 10 000 000 элементов, каждый равен 1.0.

> [!hint]- Подсказка 1: почему наивная сумма «замерзает»?
> При достижении 2²⁴ = 16 777 216 шаг между соседними f32 = 2.0. Добавление 1.0 округляется до 0, и сумма перестаёт расти.

> [!hint]- Подсказка 2: как работает Kahan summation?
> Идея: храни «компенсацию» — потерянную часть от каждого сложения. На следующей итерации добавляй компенсацию обратно.
> ```
> sum = 0, c = 0 (компенсация)
> для каждого элемента:
>   y = элемент - c        // вычитаем прошлую ошибку
>   t = sum + y            // новая сумма (с ошибкой)
>   c = (t - sum) - y      // вычисляем потерянную часть
>   sum = t
> ```

> [!hint]- Ожидаемые результаты
> - Наивная сумма: `16 777 216.0` (замёрзла на 2²⁴)
> - Kahan сумма: `10 000 000.0` (правильный ответ!)

```zig
const std = @import("std");

fn naiveSum(count: u32) f32 {
    var sum: f32 = 0.0;
    var i: u32 = 0;
    while (i < count) : (i += 1) {
        sum += 1.0;
    }
    return sum;
}

fn kahanSum(count: u32) f32 {
    var sum: f32 = 0.0;
    var c: f32 = 0.0;
    var i: u32 = 0;
    while (i < count) : (i += 1) {
        const y: f32 = 1.0 - c;
        const t: f32 = sum + y;
        c = (t - sum) - y;
        sum = t;
    }
    return sum;
}

pub fn main() void {
    const n: u32 = 10_000_000;
    std.debug.print("Наивная:  {d:.1}\n", .{naiveSum(n)});
    std.debug.print("Kahan:    {d:.1}\n", .{kahanSum(n)});
    std.debug.print("Ожидание: {d:.1}\n", .{@as(f64, @floatFromInt(n))});
}
```

## Упражнение 10: Ракета Patriot --- реальный инцидент (уровень: сложный)

В 1991 году зенитная система Patriot использовала f32 для хранения времени. Часы считали время в десятых долях секунды (целое число), потом переводили в секунды умножением на `0.1` (f32). За 100 часов работы ошибка накопилась до ~0.34 секунды --- достаточно, чтобы промахнуться мимо ракеты Scud.

**Задание:** вычисли, какая ошибка накопится при преобразовании «количество десятых секунд за 100 часов → секунды через f32».

> [!hint]- Подсказка 1
> 100 часов = 100 × 60 × 60 × 10 = 3 600 000 десятых секунд.
> Ошибка `0.1` в f32: реальное значение ≈ 0.100000001490116... (чуть больше 0.1).
> Ошибка каждой десятой = ~1.49e-8 секунды.

> [!hint]- Подсказка 2
> Наивный подход (как в Patriot): `f32(3600000) * f32(0.1)`.
> Точный подход: `f64(3600000) * f64(0.1)` или `f64(3600000) / f64(10.0)`.
> Сравни разницу.

> [!hint]- Ожидаемый результат
> Ошибка ≈ 0.0053 секунды при простом умножении. Но в реальном Patriot ошибка накапливалась иначе — через 3 600 000 отдельных умножений 0.1, что давало ~0.34 секунды. Ракета Scud летит со скоростью ~1700 м/с, за 0.34 с она пролетает ~578 м.

```zig
const std = @import("std");

pub fn main() void {
    const tenths: u32 = 3_600_000; // 100 часов в десятых секунды

    // Способ 1: как в Patriot (одно умножение на f32)
    const time_f32: f32 = @as(f32, @floatFromInt(tenths)) * 0.1;

    // Способ 2: точный (через f64)
    const time_f64: f64 = @as(f64, @floatFromInt(tenths)) * 0.1;

    // Способ 3: как в РЕАЛЬНОМ Patriot (накопление ошибки)
    var accumulated: f32 = 0.0;
    var i: u32 = 0;
    while (i < tenths) : (i += 1) {
        accumulated += 0.1;
    }

    std.debug.print("f32 (одно умножение): {d:.10} сек\n", .{time_f32});
    std.debug.print("f64 (точный):         {d:.10} сек\n", .{time_f64});
    std.debug.print("f32 (накопление):     {d:.10} сек\n", .{accumulated});
    std.debug.print("\n", .{});
    std.debug.print("Ошибка (одно умн.):   {d:.10} сек\n", .{@as(f64, time_f32) - time_f64});
    std.debug.print("Ошибка (накопление):  {d:.10} сек\n", .{@as(f64, accumulated) - time_f64});

    // При скорости Scud ~1700 м/с, промах = ошибка * 1700
    const miss = (@as(f64, accumulated) - time_f64) * 1700.0;
    std.debug.print("Промах ракеты: {d:.1} метров\n", .{miss});
}
```
