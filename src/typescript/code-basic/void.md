# void

## Зачем void, если есть undefined

В JavaScript функция без `return` возвращает `undefined`. Но в TypeScript `void ≠ undefined` — у них разная семантика при **контекстной типизации**.

## Проблема, которую решает void

```ts
const numbers = [1, 2, 3];
const result: number[] = [];

// Array.push() возвращает number (новую длину)
numbers.forEach((n) => result.push(n)); // OK
```

Колбек `forEach` типизирован как `(value: T) => void`. Если бы он требовал `=> undefined`, то `result.push(n)` дал бы ошибку компиляции, потому что `push` возвращает `number`, а не `undefined`.

**`void` означает: возвращаемое значение будет проигнорировано.** Это позволяет передавать функции, которые что-то возвращают, туда, где возврат не используется.

## Два контекста использования

### 1. Контекстная типизация (callback position)

```ts
type VoidFunc = () => void;

const f: VoidFunc = () => true; // OK — возврат игнорируется
const v = f(); // тип v — void, использовать нельзя
```

Здесь `void` — это «мне неважно, что ты вернёшь». Подходит любой тип возврата.

### 2. Явное объявление функции

```ts
function foo(): void {
  return true; // Error!
}
```

Здесь `void` — прямое обязательство: «эта функция **не должна** возвращать значение». Компилятор проверяет это строго.

## Правило

| Контекст | Поведение `void` |
|---|---|
| Тип колбека (`type Fn = () => void`) | Разрешает любой возврат, но результат не используется |
| Явная аннотация (`function f(): void`) | Запрещает `return <value>` |
