Что делает sequence в fp-ts?	"Переворачивает" вложенность: <code>Array&lt;F&lt;A&gt;&gt; → F&lt;Array&lt;A&gt;&gt;</code>. Например, <code>A.sequence(O.Applicative)</code> превращает массив Option в Option массива. Если хотя бы один элемент None/Left — весь результат None/Left. Аналог Promise.all.	fp::traversable::sequence
Что делает traverse и чем отличается от sequence?	<code>traverse = map + sequence</code> в одном шаге. <code>A.traverse(E.Applicative)(f)</code> применяет f к каждому элементу и собирает результаты. Вместо <code>pipe(arr, A.map(f), A.sequence(...))</code> пишем <code>pipe(arr, A.traverse(...)(f))</code>.	fp::traversable::traverse
Чем ApplicativePar отличается от ApplicativeSeq для TaskEither?	ApplicativePar запускает все TaskEither параллельно (как Promise.all) — быстрее. ApplicativeSeq выполняет последовательно — для rate limiting или когда порядок важен. По умолчанию используйте Par.	fp::traversable::parallel
Что такое Reader в fp-ts?	<code>Reader&lt;Env, A&gt; = (env: Env) => A</code> — ленивое вычисление, которому нужно окружение (зависимости) для получения результата. Зависимости описаны в типе, передаются при запуске. Dependency injection без фреймворков.	fp::reader
Что такое ReaderTaskEither (RTE)?	<code>RTE&lt;R, E, A&gt; = (env: R) => () => Promise&lt;Either&lt;E, A&gt;&gt;</code>. Объединяет три монады: Reader (зависимости R), Task (async), Either (ошибки E). Позволяет строить пайплайны с зависимостями, async и ошибками в одном типе.	fp::rte
Как RTE решает проблему dependency injection?	Зависимости описаны в типе R (AppEnv). Функции декларируют, что им нужно: <code>RTE&lt;AppEnv, string, User&gt;</code>. Окружение передаётся один раз на верхнем уровне. Для тестов подменяете env на мок. Никаких DI-фреймворков.	fp::rte::di
Что делает RTE.ask?	Возвращает всё окружение целиком: <code>RTE.ask&lt;AppEnv&gt;()</code> → <code>RTE&lt;AppEnv, never, AppEnv&gt;</code>. Полезно для доступа к env внутри pipe-цепочки. <code>RTE.asks(env => env.config)</code> — для получения части окружения.	fp::rte::ask
Что делает RTE.local?	Сужает или трансформирует окружение: если функции нужен <code>{ db: Db }</code>, а у вас <code>AppEnv</code>, <code>RTE.local((env: AppEnv) => ({ db: env.db }))</code> адаптирует окружение. Полезно для композиции модулей с разными зависимостями.	fp::rte::local
Когда НЕ стоит использовать RTE?	1) Зависимость одна — передайте аргументом. 2) Проект маленький (2-3 функции). 3) Команда не знакома с fp-ts. Начните с TaskEither + простые аргументы, переходите на RTE когда прокидывание зависимостей становится болезненным.	fp::rte::when-not
Как traverse помогает с валидацией массива?	<code>A.traverse(E.getApplicativeValidation(getSemigroup&lt;string&gt;()))(validateItem)</code> — валидирует каждый элемент массива и собирает ВСЕ ошибки. С обычным E.Applicative остановится на первой ошибке.	fp::traversable::validation
