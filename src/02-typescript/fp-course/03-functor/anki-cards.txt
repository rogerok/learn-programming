Что такое функтор?	Тип-конструктор F, для которого определена операция <code>map: (A => B) => (F&lt;A&gt; => F&lt;B&gt;)</code>, подчиняющаяся двум законам (identity и composition). Примеры: Array, Option, Either, Tree — любой "контейнер", к содержимому которого можно применить функцию, сохранив структуру.	fp::functor
Сформулируйте закон identity для функтора.	<code>map(x => x)(fa)</code> должен вернуть значение, идентичное <code>fa</code>. Если функция ничего не делает с содержимым, контейнер не должен измениться. Позволяет безопасно удалять <code>map(identity)</code> из цепочки.	fp::functor::laws
Сформулируйте закон composition для функтора.	<code>pipe(fa, map(f), map(g))</code> должен давать тот же результат, что <code>pipe(fa, map(x => g(f(x))))</code>. Два последовательных map можно заменить одним map со скомпонованной функцией. Позволяет безопасно объединять или разбивать шаги в pipe.	fp::functor::laws
Зачем нужны законы функтора на практике?	Закон composition гарантирует, что рефакторинг <code>pipe(fa, map(f), map(g))</code> → <code>pipe(fa, map(flow(f, g)))</code> не сломает программу. Закон identity гарантирует, что <code>map(x => x)</code> можно удалить. Без законов механические преобразования кода небезопасны.	fp::functor::laws
Как работает map для Either?	<code>map</code> применяет функцию только к значению внутри <code>Right</code>. <code>Left</code> проходит нетронутым — ошибка "пробрасывается" без вызова функции. Аналогия: <code>Right</code> = <code>Some</code>, <code>Left</code> = <code>None</code> (но с информацией об ошибке).	fp::functor::either
Что такое Either&lt;E, A&gt; и чем он отличается от Option&lt;A&gt;?	<code>Either&lt;E, A&gt; = Left&lt;E&gt; | Right&lt;A&gt;</code>. Как Option, но Left содержит причину отсутствия значения (ошибку типа E), а не просто "пусто". Right = успех, Left = ошибка с описанием. Мнемоника: Right = правильный.	fp::either
Что такое lifting в контексте функторов?	Другой взгляд на map: вместо "применить f внутри контейнера" — "поднять функцию (A => B) до функции (F&lt;A&gt; => F&lt;B&gt;)". <code>O.map(increment)</code> возвращает функцию <code>Option&lt;number&gt; => Option&lt;number&gt;</code>. Функция не знает о контейнере, но работает с ним через map.	fp::functor::lifting
Почему Promise — не корректный функтор?	<code>.then</code> автоматически разворачивает вложенные Promise (auto-flattening). Если f возвращает <code>Promise&lt;B&gt;</code>, то <code>promise.then(f)</code> даёт <code>Promise&lt;B&gt;</code>, а не <code>Promise&lt;Promise&lt;B&gt;&gt;</code>. Это нарушает закон composition: <code>.then</code> одновременно и map, и chain. В fp-ts используется Task, который разделяет эти операции.	fp::functor::promise
Что общего между Array.map, Option.map и Either.map?	Все три имеют одну сигнатуру: <code>(A => B) => (F&lt;A&gt; => F&lt;B&gt;)</code>. Все сохраняют структуру контейнера (длину массива, Some/None, Right/Left) и трансформируют только содержимое. Все подчиняются двум законам функтора. Это и есть суть абстракции "функтор".	fp::functor
Верно ли, что функтор = контейнер с данными?	Не обязательно. <code>IO&lt;A&gt;</code> хранит не значение, а функцию, которая при вызове произведёт значение. Тем не менее у IO есть корректный map. Для начала интуиция "контейнер" полезна, но она не полна — функтор это любой тип-конструктор с корректным map.	fp::functor::misconceptions
Как fp-ts предоставляет единый интерфейс map для разных контейнеров?	Каждый модуль экспортирует свою функцию map: <code>O.map</code>, <code>E.map</code>, <code>A.map</code>. Все работают одинаково через pipe: <code>pipe(value, X.map(f))</code>. Под капотом fp-ts использует type class и кодировку Higher-Kinded Types (HKT).	fp::functor::fp-ts
