# Chapter: IO в fp-ts

> [!info] Context
> `IO` в `fp-ts` моделирует **синхронное вычисление с side effects**, которое при запуске возвращает значение типа `A`. На уровне типа это просто `() => A`, но важна не форма, а договор: эффект **отложен**, его можно безопасно комбинировать через `map` / `flatMap`, а запускать нужно только на границе программы.
>
> **Зачем это нужно:** чтобы не размазывать `console.log`, `Date.now()`, `Math.random()` и другой impurity-код по всей программе.
>
> **Пререквизиты:** [[1.pipe-and-flow]], [[2.Option]], [[3.Either]], [[3.3.Either flatMap (chain)|Either chain]]

## Overview

Главная идея `IO` проста: вместо того чтобы **сразу выполнять эффект**, мы сначала создаём его **описание**.

```typescript
import * as IO from 'fp-ts/IO';

const now: IO.IO<number> = () => Date.now();
```

`now` здесь не число. Это программа, которая **может** дать число, когда её вызовут.

Полезно думать об `IO` в три шага:

1. Описать эффект как значение типа `IO<A>`
2. Скомпоновать его с другими эффектами и чистыми функциями
3. Запустить в точке входа, обработчике события, CLI-команде или тесте

```mermaid
graph LR
  A["Side effect<br/>Date.now / console.log / Math.random"] --> B["IO&lt;A&gt;<br/>отложенное описание"]
  B --> C["map / flatMap / tap<br/>композиция"]
  C --> D["program()"]
  D --> E["Реальное выполнение"]
```

В этой главе разберём:

- чем `IO` отличается от обычной функции;
- как использовать `map`, `flatMap`, `tap`, `traverseArray`;
- где запускать `IO`;
- когда нужно перейти на `IOEither` или `Task`.

**Краткий вывод:** `IO` нужен не для "магии", а для дисциплины: побочный эффект становится явным и компонуемым значением.

## Deep Dive

### 1. Что именно моделирует `IO`

По официальной модели `fp-ts`:

```typescript
export interface IO<A> {
  (): A;
}
```

Это означает:

- вычисление синхронное;
- вычисление может быть effectful или non-deterministic;
- при успешном запуске оно возвращает `A`.

Примеры подходящих эффектов:

- чтение времени через `Date.now()`;
- генерация случайного числа через `Math.random()`;
- запись в `console`;
- работа с mutable state или другими синхронными эффектами.

Важно не спутать **результат** и **описание вычисления**:

```typescript
const timestamp = Date.now();
// timestamp: number
// Эффект уже выполнен

const readTimestamp: IO.IO<number> = () => Date.now();
// readTimestamp: IO<number>
// Эффект пока только описан
```

Разница принципиальна. С `timestamp` ты уже потерял контроль над моментом выполнения. С `readTimestamp` ты можешь:

- отложить запуск;
- передать эффект как значение;
- комбинировать его с другими эффектами;
- запускать только в одном месте.

> [!important] Ментальная модель
> `IO<A>` лучше понимать как "рецепт получения `A`", а не как "контейнер с `A`". Значение появляется только после вызова `()`.

**Краткий вывод:** `IO` не устраняет side effects, а делает их явными и управляемыми.

### 2. Базовое использование: объявить и запустить

Обычно `IO` либо создают вручную, либо поднимают чистое значение через `IO.of`.

```typescript
import * as IO from 'fp-ts/IO';

const readNow: IO.IO<number> = () => Date.now();
const pureAnswer: IO.IO<number> = IO.of(42);
```

Запуск выглядит буквально как вызов функции:

```typescript
const timestamp = readNow();
const answer = pureAnswer();
```

Это делает `IO` очень простым типом, но именно из-за этой простоты легко скатиться в анти-паттерн:

```typescript
const program = readNow;

const a = program(); // эффект выполнился
const b = program(); // эффект выполнился ещё раз
```

Если эффект non-deterministic, `a` и `b` могут отличаться. Поэтому `IO` лучше воспринимать как **многоразовый рецепт**, а не как вычисленное значение.

> [!warning] Важное ограничение
> `IO` в `fp-ts` моделирует вычисление, которое **не должно падать с ошибкой**. Если внутри возможен `throw`, это уже плохая модель для `IO`; в таком случае нужен `IOEither`.

**Краткий вывод:** `IO` запускается обычным вызовом, но хороший стиль требует держать этот вызов на границе программы, а не раскидывать `()` по коду.

### 3. `map`: преобразовать результат, не запуская эффект заранее

`map` берёт чистую функцию `A -> B` и превращает `IO<A>` в `IO<B>`.

```typescript
import * as IO from 'fp-ts/IO';
import { pipe } from 'fp-ts/function';

const readNow: IO.IO<number> = () => Date.now();

const toIso = (timestamp: number): string =>
  new Date(timestamp).toISOString();

const readIsoTimestamp: IO.IO<string> = pipe(
  readNow,
  IO.map(toIso)
);
```

Здесь важно понять момент времени:

- `IO.map` **не запускает** `readNow`;
- он создаёт **новое описание** вычисления;
- эффект случится только при `readIsoTimestamp()`.

То есть:

```typescript
const program = pipe(
  readNow,
  IO.map(n => n + 1),
  IO.map(n => `ts:${n}`)
);

// До этой строки ничего не произошло
const result = program();
```

Это особенно полезно, когда хочется держать бизнес-логику чистой:

```typescript
const isEven = (n: number): boolean => n % 2 === 0;

const readParity: IO.IO<boolean> = pipe(
  () => Math.ceil(Math.random() * 6),
  IO.map(isEven)
);
```

`Math.random()` остаётся на краю, а `isEven` остаётся обычной чистой функцией.

**Краткий вывод:** `map` меняет значение, которое вернёт `IO`, но не меняет момент выполнения эффекта.

### 4. `flatMap`: следующий эффект зависит от предыдущего результата

Если следующая функция уже возвращает `IO`, `map` даст лишнюю вложенность:

```typescript
import * as IO from 'fp-ts/IO';
import { pipe } from 'fp-ts/function';

let visitsStore = 0;

const readVisits: IO.IO<number> = () => visitsStore;

const writeVisits = (n: number): IO.IO<number> => () => {
  visitsStore = n;
  return n;
};

const wrong = pipe(
  readVisits,
  IO.map(n => n + 1),
  IO.map(writeVisits)
);
// IO<IO<number>>
```

Проблема та же, что и с `Option` / `Either`: функция возвращает эффект, а `map` только добавляет ещё один слой.

Для этого нужен `flatMap`:

```typescript
const incrementVisits: IO.IO<number> = pipe(
  readVisits,
  IO.map(n => n + 1),
  IO.flatMap(writeVisits)
);
// IO<number>
```

Теперь поведение плоское:

1. прочитать текущее значение;
2. увеличить на `1`;
3. записать обратно;
4. вернуть новое число.

Полезно сравнивать сигнатуры:

```typescript
IO.map: <A, B>(f: (a: A) => B) => (fa: IO<A>) => IO<B>

IO.flatMap: <A, B>(f: (a: A) => IO<B>) => (fa: IO<A>) => IO<B>
```

> [!tip] Про `chain`
> Во многих старых статьях и видео по `fp-ts` ты увидишь `IO.chain`. В актуальном API это alias для `flatMap`. Для нового кода лучше писать `flatMap`: так намерение читается сразу.

**Краткий вывод:** `flatMap` нужен, когда следующий шаг сам effectful и зависит от результата предыдущего шага.

### 5. `tap`: выполнить побочный эффект и сохранить основное значение

Частый случай: нужно что-то залогировать, но продолжить цепочку с исходным значением. Для этого подходит `tap`.

```typescript
import * as IO from 'fp-ts/IO';
import * as Console from 'fp-ts/Console';
import { pipe } from 'fp-ts/function';

const roll: IO.IO<number> = () => Math.ceil(Math.random() * 6);

const logRoll = (n: number): IO.IO<void> =>
  Console.log(`Rolled: ${n}`);

const describeRoll = (n: number): string =>
  n === 6 ? 'critical' : 'normal';

const program: IO.IO<string> = pipe(
  roll,
  IO.tap(logRoll),
  IO.map(describeRoll)
);
```

Что произойдёт при `program()`:

1. сгенерируется число;
2. оно будет выведено в `console`;
3. цепочка продолжится с тем же числом;
4. результатом станет строка `'critical'` или `'normal'`.

Если бы здесь использовался `flatMap(logRoll)`, результат превратился бы в `IO<void>`, потому что `logRoll` возвращает `IO<void>`. `tap` специально нужен для случая "сделай дополнительный effect, но не теряй основное значение".

**Краткий вывод:** `tap` удобен для логирования, audit-действий и промежуточных write-операций, которые не должны ломать основную цепочку.

### 6. Несколько `IO` подряд: `traverseArray` и один общий запуск

Если у тебя есть массив данных и функция `A -> IO<B>`, не нужно вручную писать цикл. `IO.traverseArray` собирает массив эффектов в один эффект.

```typescript
import * as IO from 'fp-ts/IO';
import * as Console from 'fp-ts/Console';
import { pipe } from 'fp-ts/function';

const messages = ['build started', 'lint passed', 'deploy finished'];

const logAll: IO.IO<readonly void[]> = pipe(
  messages,
  IO.traverseArray(Console.log)
);
```

`logAll` здесь ещё ничего не печатает. Печать случится только при `logAll()`.

Это хороший паттерн:

- сначала собрать один `program`;
- потом один раз его интерпретировать.

Вместо:

```typescript
messages.forEach(message => Console.log(message)());
```

получается более контролируемый стиль:

```typescript
const program = IO.traverseArray(Console.log)(messages);
program();
```

**Краткий вывод:** `traverseArray` помогает не расползаться по коду множеству отдельных запусков `()`, а собирать один составной `IO`.

### 7. Где правильно запускать `IO`

Самая частая ошибка при знакомстве с `fp-ts`: объявить `IO`, а потом сразу же в середине цепочки вызвать `()`. Это ломает смысл абстракции.

Плохой стиль:

```typescript
const visits = incrementVisits(); // эффект выполнен слишком рано
const message = `Visits: ${visits}`;
```

Лучший стиль:

```typescript
const program: IO.IO<void> = pipe(
  incrementVisits,
  IO.flatMap(visits => Console.log(`Visits: ${visits}`))
);

program(); // граница мира
```

Под "границей мира" обычно понимают:

- обработчик UI-события;
- `main` в CLI;
- integration test;
- adapter-слой приложения.

Внутри доменной логики старайся хранить либо чистые функции, либо значения вида `IO<A>`, но не исполнять их.

> [!important] Практическое правило
> Чем меньше мест в проекте, где ты видишь `program()`, тем легче понимать и тестировать код.

**Краткий вывод:** хороший стиль с `IO` строится вокруг принципа "описать глубже, исполнить снаружи".

### 8. Когда `IO` уже недостаточно

`IO` подходит не для всех эффектов. У него очень узкая и полезная специализация.

#### Если вычисление может бросить исключение

Нужен `IOEither`.

```typescript
import * as IOE from 'fp-ts/IOEither';

const parseJson = (raw: string) =>
  IOE.tryCatch(
    () => JSON.parse(raw),
    reason => new Error(String(reason))
  );
```

`IO` по договору "never fails", поэтому `throw` внутри `IO` нарушает модель.

#### Если вычисление асинхронное

Нужен `Task`.

```typescript
import * as T from 'fp-ts/Task';

const readUser: T.Task<Response> = () => fetch('/api/user');
```

`IO` — только для synchronous computations. `fetch`, таймеры, файловые операции через async API, HTTP-запросы — это уже не `IO`.

#### Если эффекта может не быть

Нужен `IOOption`.

```typescript
import * as IOO from 'fp-ts/IOOption';
```

Типы рядом:

- `IO<A>`: синхронный effect, который возвращает `A` и не падает;
- `IOEither<E, A>`: синхронный effect, который может завершиться ошибкой `E`;
- `IOOption<A>`: синхронный effect, который может не дать значения;
- `Task<A>`: асинхронный effect.

**Краткий вывод:** не пытайся запихнуть все случаи в `IO`; сила `fp-ts` как раз в том, что эффекты разделены по свойствам.

### 9. Рабочий шаблон использования `IO`

Если убрать детали, практический шаблон почти всегда такой:

```typescript
import * as IO from 'fp-ts/IO';
import { pipe } from 'fp-ts/function';

let counter = 0;

const read: IO.IO<number> = () => counter;
const transform = (n: number): number => n + 1;
const write = (n: number): IO.IO<number> => () => {
  counter = n;
  return n;
};

const program: IO.IO<number> = pipe(
  read,
  IO.map(transform),
  IO.flatMap(write)
);
```

В этом шаблоне роли разделены чётко:

- `read` и `write` работают с внешним миром;
- `transform` остаётся чистой функцией;
- `program` описывает workflow;
- фактический запуск происходит отдельно.

Именно из-за этого `IO` полезен в обучении и в архитектуре: он заставляет провести границу между pure logic и effects.

**Краткий вывод:** если можешь показать pipeline как "read -> pure transform -> write", значит используешь `IO` по назначению.
