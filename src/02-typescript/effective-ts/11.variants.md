# Вариантность (Variance)

**Вариантность** описывает, как направление наследования типов переносится через контейнеры (`Array<T>`, `Promise<T>`, функции и т.д.).

---

## Иерархия для примеров

```ts
interface Entity {
  id: string;
}

interface User extends Entity {
  id: `u_${string}`;
  name: string;
}

interface CompanyUser extends User {
  company: string;
}

// CompanyUser ⊂ User ⊂ Entity
```

---

## Ковариантность — позиция `out` (производитель)

Контейнер **сохраняет** направление наследования.

> Если `User extends Entity`, то `Producer<User>` совместим с `Producer<Entity>`.

```ts
interface Producer<out T> {
  produce(): T;
}

const userProducer: Producer<User> = { produce: () => ({ id: "u_1", name: "Petya" }) };
const entityProducer: Producer<Entity> = userProducer; // OK — User ⊂ Entity
```

Тип `T` используется только **на выходе** (return). Подставить более узкий — безопасно.

---

## Контравариантность — позиция `in` (потребитель)

Контейнер **инвертирует** направление наследования.

> Если `User extends Entity`, то `Consumer<Entity>` совместим с `Consumer<User>`.

```ts
interface Consumer<in T> {
  consume(value: T): void;
}

const entityConsumer: Consumer<Entity> = { consume: (e) => console.log(e.id) };
const userConsumer: Consumer<User> = entityConsumer; // OK — Entity шире, чем User
```

Функция, принимающая `Entity`, справится с любым `User`. Обратное — нет.

---

## Инвариантность — позиция `in out`

Контейнер **не допускает** подстановку ни в каком направлении.

```ts
interface Transformer<in out T> {
  transform(value: T): T;
}

const userTransformer: Transformer<User> = { transform: (u) => u };

// const entityTransformer: Transformer<Entity> = userTransformer; // ❌
// const companyTransformer: Transformer<CompanyUser> = userTransformer; // ❌
```

Когда `T` используется и на входе, и на выходе — только точное совпадение.

---

## Практический пример: небезопасная ковариантность массивов

Мутабельные массивы в TypeScript ковариантны (хотя должны быть инвариантны):

```ts
type Animal = { id: string; name: string };
type Cat    = Animal & { name: `cat_${string}` };
type Dog    = Animal & { name: `dog_${string}` };

declare let dogs: Dog[];

const animals: Animal[] = dogs;      // OK — ковариантность
const newDog: Dog = { name: "dog_1", id: "123" };
animals.push({ name: "cat_1", id: "456" } as Cat); // No Error! ← небезопасно
// dogs теперь содержит Cat
```

`readonly` массивы — безопасно ковариантны, т.к. мутация запрещена.

## Функции: параметры контравариантны, возврат ковариантен

```ts
type Processor = (u: User) => User;

function processUser(u: Entity): CompanyUser {
  // u — Entity (шире User) → контравариантность параметра ✓
  return { id: "u_1", name: "Petya", company: "Dummy" };
  // CompanyUser — подтип User → ковариантность возврата ✓
}

const p: Processor = processUser; // OK
```

### Почему `u.id = '8'` не вызывает ошибки?

```ts
function processUser(u: Entity): CompanyUser {
  u.id = "8"; // Entity.id — string, присвоение string — OK по типу
  // ...
}

const petya: User = { id: "u_1", name: "Petya" };
processUser(petya);
// petya.id теперь равен "8", хотя тип User требует `u_${string}`
```

Объект передаётся **по ссылке**. Внутри функции переменная `u` имеет тип `Entity`, поэтому `u.id = "8"` валидно для компилятора. Но физически это тот же объект в памяти — инвариант `User` нарушен в рантайме.

## Модификаторы вариантности (TypeScript 4.7+)

`in` / `out` — явные аннотации вариантности на параметрах дженерика:

```ts
interface Producer<out T>     { produce(): T }         // ковариантен
interface Consumer<in T>      { consume(v: T): void }  // контравариантен
interface Transformer<in out T> { transform(v: T): T } // инвариантен
```

Преимущества явных аннотаций:
- Намерение читаемо в сигнатуре типа.
- Компилятор не анализирует тело — быстрее.
- Ошибка, если реализация нарушает заявленную вариантность:

```ts
interface BrokenProducer<out T> {
  consume(value: T): void; // ❌ Type 'T' is 'out' but used in 'in' position
}
```
