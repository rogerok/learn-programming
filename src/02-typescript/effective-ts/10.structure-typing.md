# Структурная типизация

TypeScript использует **структурную** (duck typing) типизацию: совместимость типов определяется по **форме** (составу полей), а не по имени типа.

---

## Когда структурная типизация отключается

В классах с **приватными полями** структурная типизация превращается в **номинальную**: два класса считаются несовместимыми, даже если их поля идентичны.

```ts
class User {
  private type: string = "user";
}

class Admin {
  private type: string = "user";
}

const x: User = new User();
const b: Admin = x; // ❌ Error — разные классы, разные private
```

---

## Совместимость функций с разным числом перегрузок

TypeScript при проверке совместимости функций ищет **хотя бы одну перегрузку**, которая удовлетворяет требованиям.

```ts
const EMPTY_REF = Symbol("empty ref");

type MaybeEmptyRefCreator = {
  (): typeof EMPTY_REF;
  <T>(x: T): { value: T };
};

type TransparentWrapper = <A>(x: A) => A;

declare const refWrapper: MaybeEmptyRefCreator;

const wrap: TransparentWrapper = refWrapper;
// Ошибки нет — TS нашёл перегрузку <T>(x: T): { value: T },
// которая считается совместимой с <A>(x: A) => A
```

При этом сигнатура `<A>(x: A) => A` (принимает один аргумент) считается **совместимой** с `(): typeof EMPTY_REF` (принимает ноль аргументов).

---

## Почему функция с аргументом совместима с функцией без аргументов?

Это следствие **контравариантности** параметров функций:

- Функция с одним аргументом может быть вызвана без него (аргумент проигнорируется).
- Функция без аргументов строже — она принимает "ничего".
- Поэтому `(x: A) => A` шире, чем `() => ...` — и совместима.

На практике это позволяет передавать `Array.prototype.forEach` обратный вызов с одним аргументом в контекст, где ожидается вызов без аргументов.

---

## Ключевое правило

> Для совместимости функций TypeScript проверяет **только сигнатуру** (без тела), и достаточно **одной подходящей перегрузки**.
