# Совместимость типов (Type Compatibility)

## Пример: иерархия через union

```ts
type ActiveDeal  = { status: "ok" } | { status: "at_risk" } | { status: "payment_due" };
type ClosedDeal  = { status: "completed" } | { status: "stopped" };
type Deal        = ActiveDeal | ClosedDeal;
type Lead        = { status: "contact" } | { status: "presale" };
type Entity      = Deal | Lead;

// Иерархия:
// ActiveDeal ⊂ Deal ⊂ Entity
```

---

## Processor и вариантность

```ts
type Processor<Input extends Entity, Output extends Entity> = (input: Input) => Output;

type DealProcessor = Processor<Deal, Deal>;

const x: DealProcessor = (input: Deal): Deal => input;
```

`DealProcessor` — функция, которая принимает `Deal` и возвращает `Deal`.

Важно: функция совместима с `DealProcessor`, если:
- Параметр **контравариантен**: функция может принять **более широкий** тип (`Entity`), но не более узкий (`ActiveDeal`).
- Возвращаемое значение **ковариантно**: функция может вернуть **более узкий** тип (`ActiveDeal`), но не более широкий (`Entity`).

```ts
// OK — параметр шире (Entity), возврат уже (ActiveDeal)
const y: DealProcessor = (input: Entity): ActiveDeal => ({
  status: "ok"
});

// ❌ — параметр уже (ActiveDeal) — нарушает контравариантность
const z: DealProcessor = (input: ActiveDeal): Deal => input;
```

---

## Почему контравариантность параметров?

Если `DealProcessor` обещает принять `Deal`, то реализация должна уметь работать хотя бы с `Deal`.
Функция, принимающая только `ActiveDeal`, не справится с `ClosedDeal` — и это нарушение контракта.

Функция, принимающая `Entity` (шире) — справится с любым `Deal`, поэтому она **совместима**.

---

## Совместимость функций: общее правило

```
(A) => B  совместима с  (C) => D
если:
  C ⊆ A  (параметр: тип ожидания уже реализации — контравариантность)
  B ⊆ D  (возврат: реализация уже ожидания — ковариантность)
```

Мнемоника: **"принимай шире, возвращай уже"**.
