# Type Inference & Type Refinement

## Вывод типов (Type Inference)

Когда переменная объявляется без явного типа, TypeScript выводит **достаточно узкий** тип:

```ts
const a = 42;  // тип: 42 (literal)
let b = 42;    // тип: number (расширяется, т.к. let может меняться)
```

---

## Type Refinement

**Refinement** — это процесс, когда TypeScript анализирует control flow и **отсекает из union** те варианты, которые не могут произойти в текущей ветке выполнения.

Механизмы сужения:
- `typeof`
- `instanceof`
- проверки `null` / `undefined`
- оператор `in`
- custom type guard
- литеральное сравнение (дискриминатор)

---

## Discriminated Unions

Это union, в котором каждый вариант имеет **поле-дискриминатор** — литеральное поле, уникальное для каждого члена.

```ts
type Event = { type: 'login'; user: string } | { type: 'logout'; timestamp: string }
```

TypeScript по значению `type` понимает, с каким вариантом работаем:

```ts
function handle(e: Event) {
  if (e.type === 'login') {
    console.log(e.user)     // e: { type: 'login'; user: string }
  } else {
    console.log(e.timestamp) // e: { type: 'logout'; timestamp: string }
  }
}
```

---

## Exhaustive Check

Паттерн для гарантии, что все варианты union обработаны.

```ts
declare function assertNever(x: never): void;

function handle(e: Event) {
  if (e.type === 'login') {
    console.log(e.user)
  } else if (e.type === 'logout') {
    console.log(e.timestamp)
  } else {
    assertNever(e); // если все варианты обработаны — e имеет тип never ✅
  }
}
```

Если в `Event` добавить новый вариант:

```ts
type Event =
  | { type: 'login'; user: string }
  | { type: 'logout'; timestamp: string }
  | { type: 'payment'; amount: number }; // ← новый вариант
```

...то в `else`-ветке `e` будет иметь тип `{ type: 'payment'; amount: number }`, а не `never` — ошибка компиляции укажет, что вариант не обработан.

## Безопасность методов сужения

### Безопасные

| Метод | Почему безопасен |
|---|---|
| `typeof` | Проверяет реальный рантайм-тип |
| Поле-дискриминатор | Литеральное значение уникально |
| `instanceof` | Проверяет прототипную цепочку |
| Литеральное сравнение | Значение зафиксировано |
| Custom type guard | Явная логика (если написана верно) |

### Небезопасные

| Метод | Почему небезопасен |
|---|---|
| `in` | Структурная типизация допускает лишние поля |
| По не-дискриминаторному полю | Поле может присутствовать у нескольких вариантов |

### Пример: опасность `in`

```ts
type InternalUser = { name: string; department: string; subdepartment: string };
type ExternalUser = { name: string; company: string };
type User = InternalUser | ExternalUser;

function show(user: User) {
  if ('department' in user) {
    console.log(user.subdepartment.toUpperCase()) // предполагаем InternalUser
  } else {
    console.log(user.company)
  }
}

const obj = { name: 'John', company: 'Doe', department: 'S' };
const user: ExternalUser = obj; // OK — структурная типизация

show(user); // ❌ RuntimeError: Cannot read properties of undefined (reading 'toUpperCase')
```

`obj` имеет поле `department`, поэтому `in` ошибочно считает его `InternalUser`, но `subdepartment` у него нет.
