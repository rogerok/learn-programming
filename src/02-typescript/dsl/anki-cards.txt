Что такое Fluent Builder DSL?	Паттерн проектирования API, в котором методы возвращают this, образуя цепочку вызовов, читаемую как предложение на языке предметной области. Это разновидность internal DSL.	dsl::fluent-builder
Какой главный механизм делает цепочку вызовов возможной?	Каждый метод, изменяющий состояние builder'а, возвращает this. Это позволяет вызвать следующий метод прямо на результате предыдущего: builder.from("users").where("age", ">", 18).build()	dsl::fluent-builder::mechanism
Почему в TypeScript нужно указывать return type как this, а не как имя класса?	Polymorphic this type автоматически сужается до конкретного подкласса при наследовании. Если указать имя базового класса, методы подкласса будут недоступны в цепочке после вызова метода из базового класса.	dsl::fluent-builder::typescript
Чем internal DSL отличается от external DSL?	Internal DSL живёт внутри host-языка (TypeScript) и использует его синтаксис (методы, классы). External DSL имеет собственный синтаксис и парсится отдельным парсером (например, .prisma, .graphql файлы).	dsl::classification
Что такое terminal operation в Fluent Builder?	Метод, завершающий цепочку и возвращающий результат: .build() → строку/объект, .send() → Promise<Response>, .parse() → типизированные данные. Это граница между фазой конфигурации и фазой выполнения.	dsl::fluent-builder::terminal
В чём проблема мутабельного builder'а при ветвлении?	Если создать base builder и затем вызвать .where() для одного варианта, base тоже мутируется. Второй вариант на основе base получит чужие условия. Решение — immutable builder, который возвращает новый экземпляр на каждом шаге.	dsl::fluent-builder::immutable
Как работает immutable builder?	Каждый метод возвращает новый экземпляр builder'а с обновлённым состоянием, вместо мутации this. Конструктор принимает все поля как readonly параметры. Позволяет безопасно ветвить цепочки.	dsl::fluent-builder::immutable
Что такое phantom type и зачем он в Schema DSL?	Phantom type — свойство, объявленное через declare, которое не существует в рантайме, но участвует в выводе типов. В Zod declare readonly _output: T позволяет z.infer<typeof Schema> извлечь TypeScript-тип из значения схемы.	dsl::schema::phantom-types
Как Knex.js перегружает метод .where()?	.where() принимает разные формы: .where(column, value) → равенство, .where(column, op, value) → оператор, .where(object) → несколько AND, .where(callback) → группировка. TypeScript моделирует это через function overloads.	dsl::knex::overloading
Как phantom types делают builder type-safe?	Каждый метод возвращает builder с расширенным типом состояния через intersection: State & HasTable. Terminal operation доступна только при определённом типе State. Вызов .build() без обязательных шагов даёт ошибку компиляции.	dsl::fluent-builder::type-safe
Когда Fluent Builder DSL избыточен?	Когда у объекта 2-4 обязательных поля — проще использовать объектный литерал или конструктор. Для трансформации данных лучше Pipeline DSL (pipe/map/filter). Для логических условий — Rule DSL (.when()/.check()).	dsl::fluent-builder::tradeoffs
Что такое композиция методов в Fluent Builder? Приведите пример.	Высокоуровневый доменный метод вызывает низкоуровневый: .bearer(token) внутри вызывает return this.header("Authorization", "Bearer " + token). Пользователь работает на удобном уровне абстракции, а реализация переиспользует базовые методы.	dsl::fluent-builder::composition
Почему при проектировании DSL показывают call site до реализации?	Принцип outside-in: DSL проектируется от пользовательского кода. Сначала пишут желаемый синтаксис вызова, потом строят реализацию так, чтобы этот синтаксис работал. Это гарантирует, что API удобен для потребителя, а не для автора.	dsl::design-principle
Какое правило именования методов в DSL?	Методы должны называться доменными словами (.from(), .where(), .select()), а не словами реализации (.setTable(), .addCondition(), .pushField()). DSL читается как язык предметной области.	dsl::naming
Назовите 4 реальных примера Fluent Builder DSL в экосистеме JS/TS.	1) Knex.js — Query Builder для SQL. 2) Zod — Schema DSL с выводом типов. 3) Express — Routing DSL (app.get(), app.post()). 4) jQuery — цепочки манипуляций с DOM (исторически первый массовый Fluent API в JS).	dsl::real-world
