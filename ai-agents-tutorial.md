## 1. Ментальная модель: понимание ограничений LLM

Прежде чем браться за инструменты, нужно понять, с чем мы имеем дело. LLM — это не "младший разработчик", а нечто более странное и ограниченное. Если этого не учитывать, вы будете бороться с природой модели вместо того, чтобы работать с ней.

### Smart zone vs dumb zone

Контекстное окно у Opus/Sonnet 4.6 — 200 000 токенов (или 1 миллион в новых версиях). Но это не значит, что весь объём одинаково "умный".

| Зона | Диапазон | Поведение |
|---|---|---|
| Smart zone | первые ~80 000 токенов (40% от 200k) | модель работает чётко, удерживает контекст |
| Dumb zone | 80–90% и выше | внимание начинает распадаться, рассуждения деградируют |

Внимание у трансформеров масштабируется квадратично: каждый новый токен должен "связаться" с каждым уже имеющимся. Поэтому увеличение лимита контекста до 1 миллиона по факту просто добавляет ещё больше "глупой зоны" — smart zone остаётся примерно тем же.

**Практический вывод:** держите рабочую сессию в районе ниже 40% контекста. Когда вы видите 35–40% — пора задуматься о `/clear`.

### Statelessness (отсутствие памяти)

Каждая новая сессия — чистый лист. Агент — это как новый разработчик, приходящий в команду каждые 20 минут без знания истории проекта. Это значит, что **качество вашего кодбейза напрямую определяет эффективность агента**: насколько быстро он может сориентироваться, насколько связаны логически смежные куски кода, есть ли быстрые feedback loops.

### Hallucinations (галлюцинации)

Это не баги, а закономерность. Когда вы спрашиваете LLM о чём-то за пределами его сжатого знания, модель так хочет помочь, что заполняет пробелы правдоподобной выдумкой: несуществующие методы, неверная структура API, выдуманные имена пакетов. Лекарство — давать документацию и примеры в контекст, а не полагаться на "память" модели.

### Knowledge cutoff

Модель обучена до определённой даты. Всё, что вышло позже — невидимо. Если ваша библиотека выпустила мажорный релиз после cutoff, модель про него не знает. Опять же: подавайте документацию в контекст.

## 2. Базовые инструменты Claude Code

### Запуск и базовые команды

```bash
claude                  # запустить Claude Code в текущем каталоге
claude resume           # вернуться к последней сессии
claude --continue       # быстрый шорткат на продолжение последней сессии
```

### Внутри сессии

| Команда | Назначение |
|---|---|
| `/model` | выбрать модель (Opus 4.6 для 5X Max, Sonnet 4.6 для Pro; Haiku не использовать) |
| `/clear` | полная очистка контекста — основной инструмент дисциплины |
| `/context` | посмотреть, чем сейчас заполнен контекст |
| `/usage` | сколько потрачено по подписке |
| `/compact` | сжать историю в краткое summary (используется редко, см. ниже) |
| `/resume` | список прошлых сессий в этом репозитории |
| `/skills` | список доступных skills |
| `/memory` | подгрузить изменения CLAUDE.md в текущую сессию без перезапуска |

### Ввод и навигация

- `@filename` — вставить файл в контекст с автодополнением. Это основной способ давать агенту нужный контекст без ручного копирования.
- `Shift+Tab` — переключение между режимами: execute → plan mode → accept-edits.
- `Esc` — прервать текущую генерацию.
- `Esc Esc` — режим rewind: можно вернуться к любой точке диалога.
- `Ctrl+B` — увести запущенный процесс (например, dev server) в фон.
- `Ctrl+Z` → `fg` — приостановить Claude Code и взять терминал под свой контроль (что вы ввели — агент не увидит), затем вернуть управление агенту.
- `Ctrl+C Ctrl+C` — выйти из Claude Code (сессия сохраняется).

### Bash из сессии

```
! pnpm typecheck     # быстрая команда: вывод сразу попадает в контекст агента
pnpm dev             # длинный процесс — потом Ctrl+B чтобы свернуть в фон
```

Префикс `!` — для быстрых команд, результат которых должен увидеть агент. Без префикса — обычный запуск с возможностью свернуть в фон.

### IDE integration

Claude Code умеет интегрироваться с VS Code и аналогами. Главная польза — diff'ы предлагаемых изменений показываются прямо в редакторе с подсветкой синтаксиса, а не в терминале. Можно принять изменения кнопкой "Accept Proposed Changes" или вручную подправить файл и сохранить — это считается принятием.

### Status line с процентом контекста

По умолчанию Claude Code не показывает, сколько контекста съедено. Установите [ccstatusline](https://www.npmjs.com/package/ccstatusline):

```bash
mkdir -p ~/.config/ccstatusline
```

В `~/.config/ccstatusline/settings.json` положите конфиг с виджетами `context-length` и `context-percentage` (рабочий пример есть в исходниках курса).

В `~/.claude/settings.json` добавьте:

```json
{
  "statusLine": {
    "type": "command",
    "command": "npx ccstatusline@latest"
  }
}
```

После рестарта вы увидите строку вида `186.2k (17.3%)` — без неё дисциплина по контексту трудно реализуется.

### Permissions (права)

Claude Code по умолчанию строг к командам. При первом обращении к незнакомой команде вы увидите три варианта: разрешить разово, разрешить всегда, отклонить. Все разрешения хранятся в `.claude/settings.local.json`:

```json
{
  "permissions": {
    "allow": ["Bash(pnpm *)", "Bash(gh issue create *)"],
    "deny":  ["Bash(git push *)"]
  }
}
```

Шаблоны с `*` экономят время. Опасные команды кладите в `deny`. Если хотите шарить разрешения с командой — переименуйте `settings.local.json` в `settings.json` и закоммитьте.

### Sub-agents

Claude Code активно делегирует подзадачи sub-агентам — например, для exploration. У каждого sub-агента свой контекст; в основной (orchestrator) возвращается только summary. Это механизм экономии контекста главной сессии — благодаря ему orchestrator дольше остаётся в smart zone. На практике вы заметите это в виде записей вроде:

```
● Explore(Explore codebase for PRD) Sonnet 4.6
  ⎿  Done (28 tool uses · 78.0k tokens · 1m 38s)
```

78k токенов "сгорели" в sub-агенте, но в главный контекст вернулось только сжатое summary.

## 3. Управление контекстом и сессией

### Plan mode

Plan mode — стрипнутая версия execute mode: агент может только **читать** файлы, **запускать bash-скрипты** и **обращаться к MCP-серверам**, но не может **писать** файлы. Плюс системный промпт настроен на сбор требований и создание плана.

Зачем:
- Позволяет вам уточнить требования через диалог. Вы стартуете с одного предложения, а LLM "интервьюирует" вас, докапываясь до спорных решений.
- Заставляет агента сначала исследовать кодбейз и убедиться, что он понимает проект, прежде чем предложить решение.

Включить: `Shift+Tab` пока внизу не появится `plan mode on`.

### Rewind (откат)

`Esc Esc` открывает историю диалога. Можно выбрать точку и решить, что восстановить:

| Вариант | Эффект |
|---|---|
| Restore code and conversation | полный откат к моменту |
| Restore conversation only | контекст откатывается, код остаётся |
| Restore code only | код откатывается, контекст остаётся |

Полезно при экспериментах: попробовали что-то — откатились, попробовали иначе.

### `/clear` vs `/compact`

`/clear` — полностью обнуляет контекст. Это **основной инструмент дисциплины**.

`/compact` — пытается ужать историю в summary через LLM. Звучит привлекательно, но на практике у компакта есть существенные минусы:

- Создаёт "осадок" в контексте: каждый компакт оставляет нечёткое воспоминание о прошлой сессии, и эти осадки накапливаются непредсказуемо.
- Тратит токены и время.
- При сравнимых усилиях `/clear` + повторное explore часто даёт более предсказуемый результат.

Когда `/compact` всё-таки полезен:
- После большой имплементации хочется сделать QA, и не хочется терять весь контекст обсуждения.
- Сложный отладочный сеанс, где нельзя позволить себе всё забыть.

При `/compact` дайте инструкцию: *"я только что закончил имплементацию фичи и хочу сделать QA"* — это сильно улучшает качество summary.

### Когда чистить контекст

Простое правило, выработанное практикой:

- 0–30% — можно работать спокойно
- 30–40% — пора задуматься о завершении задачи или коммите
- 40%+ — вы в "сумеречной зоне", качество ответов начинает падать
- Перед каждой новой задачей — `/clear`

## 4. Steering: как направлять агента

Steering — это всё, что заставляет агента следовать вашим конвенциям и предпочтениям без необходимости повторять их в каждом промпте.

### CLAUDE.md (он же AGENTS.md)

Это README для агента: файл в корне проекта, который автоматически подгружается в каждую сессию.

```bash
touch CLAUDE.md
```

Положите туда правило, например:

```
When you have a function with more than one parameter with the same type,
use an object parameter instead of positional parameters:

// BAD
const addUserToPost = (userId: string, postId: string) => {};

// GOOD
const addUserToPost = (opts: { userId: string; postId: string }) => {};
```

Чтобы изменения CLAUDE.md подхватились в **текущей** сессии (без рестарта), используйте `/memory`.

### Проблемы CLAUDE.md

1. **Глобальность.** Содержимое CLAUDE.md идёт в каждый запрос — это токены, потраченные на каждой итерации. CLAUDE.md в 1000 строк съедает 5–10% контекста ещё до того, как вы что-то спросили.

2. **Конкуренция инструкций.** Чем больше правил, тем сильнее они конкурируют между собой и с конкретным промптом.

3. **Claude часто игнорирует CLAUDE.md.** Системный промпт Claude Code содержит формулировку: *"This context may or may not be relevant to your tasks. You should not respond to it unless it's highly relevant."* Это значит, что steering через CLAUDE.md **ненадёжен**.

4. **Команда `claude init` создаёт мусор.** Она заполняет CLAUDE.md очевидными вещами вроде содержимого `package.json`, которое модель и так найдёт за секунду. **Не запускайте `claude init`** — будете тратить токены на ненужное.

### Что класть в CLAUDE.md

Только то, что действительно нужно агенту в **каждой** сессии и что **не очевидно** из самого кода. Хорошие кандидаты:

- Конвенция именования (например: "файлы с суффиксом `-service.ts` должны иметь рядом файл `.test.ts`")
- Узкие предпочтения по API (объектные параметры вместо позиционных)
- Названия скриптов package.json **только если** они нестандартные

Плохие кандидаты:
- Подробные инструкции по работе с конкретным фреймворком (это уйдёт в Skill)
- Документация библиотек
- Списки файлов и структура проекта

### Progressive disclosure: ключевой принцип

Идея из UX-дизайна: вместо того чтобы вывалить всю информацию сразу, давайте пользователю карту и позволяйте навигироваться по ней.

Применительно к агенту:

- Вместо CLAUDE.md на 1000 строк со всеми инструкциями — CLAUDE.md с короткими ссылками на отдельные файлы.
- Инструкции для работы с React Router живут в одном месте, для базы данных — в другом, для фронтенда — в третьем.
- Агент видит только metadata (заголовок и описание) и подгружает нужную пачку **только когда она ему действительно нужна**.

Этот принцип воплощён в Skills — следующий раздел.

## 5. Skills: ключевой механизм работы с агентом

Agent Skills — открытый формат для расширения возможностей агентов. Поддерживается Claude Code и другими (Cursor, Codex). Skill — это папка с инструкциями, скриптами и ресурсами, которые агент может **обнаружить** (но не обязан использовать).

### Структура Skill

Skill — это папка, обычно по пути:

```
.claude/skills/<skill-name>/SKILL.md     # внутри проекта (шарится с командой)
~/.claude/skills/<skill-name>/SKILL.md   # глобально для пользователя
```

Минимальный SKILL.md:

```markdown
---
name: pnpm-not-found
description: Fix pnpm command not found errors by enabling corepack. Use this when pnpm cannot be found, corepack errors appear or the package manager is missing.
---

If you encounter `pnpm: command not found`, run:

corepack enable

This wires up pnpm as a managed package manager.
```

Front matter — это "витрина" skill'а: агент видит только `name` и `description`, и решает, нужно ли подгрузить тело SKILL.md в контекст.

### Типы Skills

| Тип | Кто вызывает | Использование |
|---|---|---|
| Agent-invoked | сама модель | помощь в задачах: правила, гайдлайны, фиксы редких ошибок |
| User-invoked | человек через `/skill-name` | смена поведения агента: написать PRD, запустить TDD-цикл |

Управляется через front matter:

```yaml
disable-model-invocation: true   # только пользователь
user-invocable: false            # только модель
```

### Прогрессивная глубина внутри skill'а

Внутри SKILL.md можно ссылаться на другие файлы — скрипты, доп. документацию, картинки. Например:

```markdown
For detailed migration steps see [migration script](./script.sh).
```

LLM спокойно открывает такие ссылки. Так можно "спрятать" большую инструкцию или скрипт, и она подгрузится только если действительно понадобится.

### Готовые skills, которые стоит знать

В сообществе сложился набор полезных skills. Установка через `npx skills add <author>/<repo>/<skill>`:

| Skill | Назначение |
|---|---|
| `/grill-me` | агент допрашивает вас по поводу плана/идеи, пока не достигнут общего понимания |
| `/write-a-skill` | создаёт новый skill (skill для написания skills) |
| `/write-a-prd` | пишет полноценный PRD с user stories, делая допросы по ходу |
| `/prd-to-plan` | разбивает PRD на multi-phase план с tracer bullets |
| `/prd-to-issues` | разбивает PRD на отдельные GitHub issues с зависимостями (Kanban) |
| `/do-work` | управляет циклом "понять → сделать → проверить → закоммитить" |
| `/improve-codebase-architecture` | находит места в кодбейзе, где можно углубить модули |

Установка примеров (Matt Pocock):

```bash
npx skills add mattpocock/skills/grill-me
npx skills add mattpocock/skills/write-a-skill
```

После установки — `/skills` в Claude Code показывает доступные.

### Skill для своих специфичных проблем

Skills особенно ценны для **редких, но болезненных** случаев. Пример из реальной жизни: на одном проекте собрался `better-sqlite3`, и при переключении между средами (локально → Docker) ругался на mismatch версий нативного модуля. Решение — `pnpm rebuild`. Кладём это в skill:

```markdown
---
name: better-sqlite-rebuild
description: Use when seeing errors about better-sqlite3 native module version mismatch
user-invocable: false
---

Run `pnpm rebuild better-sqlite3` to rebuild the native module against the current Node version.
```

Теперь, попадая в эту ошибку, агент сам найдёт skill и сделает правильную вещь — без того чтобы вы держали это правило в голове.

## 6. Воркфлоу для нового проекта

Здесь описан цикл, по которому стоит входить в новый проект — независимо от того, начинаете ли вы с нуля или поднимаете чужой репозиторий.

### Шаг 1. Понять требования

Стартуйте не с кода, а с понимания. Воспользуйтесь skill'ом `/grill-me` или просто включите plan mode и опишите идею в одном-двух предложениях:

```
Make it so that students can comment on lessons and instructors can moderate their comments.
```

Дальше LLM начнёт допрашивать вас:
- Должны ли комментарии редактироваться?
- Soft delete или hard delete?
- Кто видит комментарии — все или только подписанные?
- Плоский список или деревья?
- Лимит символов?

Отвечайте подробно. Один такой "допрос" заменяет несколько часов созвонов с командой о деталях.

### Шаг 2. Написать PRD

Используйте `/write-a-prd`. Skill сам проведёт через генерацию документа: проблема, решение, user stories с нумерацией, technical decisions, out-of-scope.

PRD — это **карта пункта назначения**, а не маршрута. В нём фиксируются "что" и "почему", но не "как именно реализовать". Хорошая структура PRD:

- **Problem statement** — почему фича нужна вообще
- **Solution overview** — на верхнем уровне что мы строим
- **User stories** (нумерованные) — "как роль X я хочу Y чтобы Z"
- **Modules / interfaces** — какие модули будут, какие интерфейсы изменятся
- **Technical decisions** — конкретные решения (например, "сортировка таблицы — на клиенте")
- **Schema changes** — изменения базы
- **Out of scope** — что мы НЕ делаем (важно для последующих PRD)

Нумерация user stories критична: на стадии плана и имплементации можно ссылаться через `#3` или `US-3`.

### Шаг 3. Разбить на задачи: план или Kanban

#### Вариант A — Multi-phase план (для одного человека)

`/prd-to-plan` создаёт markdown-файл `plans/<feature>.md` с фазами. Ключевое требование к плану — **tracer bullets**, или **vertical slices**.

| Подход | Что делается в каждой фазе |
|---|---|
| Horizontal (плохо) | один слой целиком: фаза 1 — backend, фаза 2 — UI, фаза 3 — роуты |
| Vertical slices (хорошо) | каждая фаза затрагивает все слои: фаза 1 — минимальный сквозной кейс end-to-end, фазы 2-N — расширения |

Tracer bullet — это термин из Pragmatic Programmer (по аналогии с трассирующей пулей: артиллерист видит, куда летят снаряды, и корректирует прицел). Применительно к коду: после первой фазы вы должны иметь **рабочий end-to-end путь** на минимальных данных. Это ловит unknown-unknowns на ранней стадии.

LLM знает, что такое tracer bullets — достаточно сказать `use tracer bullets` в промпте.

#### Вариант B — Kanban через GitHub Issues (для команды и AFK-режима)

`/prd-to-issues` ломает PRD на независимые задачи и публикует их как GitHub issues с описанием зависимостей ("blocked by #5"). У этого подхода преимущества:

- **Параллелизм:** несколько разработчиков (или вы и агент) могут брать независимые задачи одновременно.
- **Бесконечная расширяемость:** добавить QA-замечание = создать новый issue.
- **Естественный QA:** один issue — отдельный testable объект.

Каждый issue получает метку: AFK (можно делать автономно) или HITL (human-in-the-loop, требует решений человека). Skill также генерирует финальный QA issue с чеклистом.

### Шаг 4. Настроить feedback loops

До начала имплементации — настройте:

1. **TypeScript strict mode** или аналог для вашего языка — самый простой и сильный feedback loop.
2. **Тестовый раннер** — Vitest, Jest, что угодно. Команда вида `pnpm test`.
3. **Pre-commit hooks через Husky + lint-staged**:
   - форматтер на staged-файлах (Prettier);
   - typecheck;
   - тесты.

Установка хука: можно дать Claude Code задачу "implement pre-commit hooks with Husky, lint-staged, type-check and tests" — он сам поставит зависимости и сгенерирует `.husky/pre-commit`. Либо вручную:

```bash
pnpm dlx husky init
echo "pnpm exec lint-staged && pnpm typecheck && pnpm test" > .husky/pre-commit
```

После этого **агент не сможет закоммитить сломанный код** — pre-commit отвергнет коммит, и в feedback вернётся текст ошибки, который агент тут же будет править.

Подробнее — в разделе [Feedback loops](#9-feedback-loops-обратная-связь-как-опора-качества).

### Шаг 5. Минимальный CLAUDE.md

Положите в корень `CLAUDE.md` с короткими правилами, специфичными для проекта. Не более 30–50 строк. Например:

```markdown
# Conventions

- Functions with multiple same-typed parameters use object params: `(opts: { userId: number; postId: number })`
- Files matching `*-service.ts` must have an accompanying `*-service.test.ts`
- Use Drizzle ORM for all DB access; do not write raw SQL
- Date fields are stored as Unix timestamps in seconds (not ms)
```

### Шаг 6. Имплементация — цикл Plan → Execute → Clear

См. [раздел 8](#8-цикл-plan--execute--clear).

## 7. Воркфлоу для существующего проекта

Если проект уже большой и существует давно, подход немного другой.

### Шаг 1. Дать агенту понять кодбейз

Откройте Claude Code в корне репозитория и попросите:

```
Tell me what the tech stack of this repo is and what its intended purpose is.
```

Дальше — серия наводящих вопросов, чтобы вы и агент **одновременно** строили карту проекта:

```
What user roles and permission levels exist in this application?
Where is the authentication and authorization logic?
What does the database schema look like?
How does routing work — what mode/library is used?
Which modules are tested, and at what boundary?
```

Агент будет читать файлы, иногда запускать `tree` или `ls`, и собирать представление. Полезный побочный эффект: вы тоже получаете быстрый онбординг в чужой код.

### Шаг 2. Найти узкие места архитектуры

Запустите `/improve-codebase-architecture`. Skill пройдётся по проекту и предложит несколько кандидатов на углубление модулей. Например:

> **Quiz subsystem.** Three services (quizScoring, quizCRUD, xpAwarding) могут быть объединены в один deep module. Текущий quizScoring использует raw BetterSQLite3, остальные — Drizzle ORM. Это inconsistency и потенциальный баг.

Skill также сгенерирует несколько альтернативных дизайнов интерфейса и порекомендует один. На выходе — GitHub issue с RFC рефакторинга, который можно делать постепенно.

### Шаг 3. Настроить минимальный CLAUDE.md и Skills под проект

Если вы видите, что агент копирует "плохие паттерны" из существующего кода (а он будет это делать — кодбейз для агента это самый сильный источник истины), напишите явные правила в CLAUDE.md и/или skill для частых граблей.

Пример из практики: агент склонен использовать позиционные параметры, потому что в проекте так пишут. Добавляем в CLAUDE.md правило про объектные параметры — следующая сессия пишет уже правильно.

### Шаг 4. Поднять feedback loops до состояния "годно"

Если в проекте слабая тестовая обвязка — это критично. Агент будет работать тем хуже, чем меньше у него способов проверить себя. Минимум:

- typecheck должен проходить локально и быть быстрым (< 30 сек);
- тесты должны быть репрезентативными — даже базовое покрытие сильно помогает агенту;
- pre-commit hooks (см. выше).

### Шаг 5. Поэтапное внедрение фич

Дальше — обычный цикл `Plan → Execute → Clear` (раздел 8), но с особенностями:

- На стадии плана **сначала** дайте агенту просмотреть существующие модули — чтобы новый код встроился в сложившиеся паттерны.
- Используйте `@<file>` чтобы подложить релевантные куски кодбейза в контекст явно.
- На каждом шаге проверяйте, не "убежал" ли агент от ваших конвенций.

## 8. Цикл Plan → Execute → Clear

Это базовый ритм работы с агентом, который выработался у большинства продвинутых пользователей.

### Описание цикла

```
┌─────────────────────────────────────────────┐
│ 1. PLAN MODE                                │
│    - Опишите задачу одним абзацем           │
│    - Shift+Tab → plan mode on               │
│    - LLM исследует код, задаёт вопросы      │
│    - На выходе — план в markdown            │
├─────────────────────────────────────────────┤
│ 2. EXECUTE                                  │
│    - Shift+Tab → выходим из plan mode       │
│    - Опционально: clear если контекст > 35% │
│    - Агент имплементирует, прогоняет тесты  │
│    - Вы наблюдаете и корректируете при нужде│
├─────────────────────────────────────────────┤
│ 3. QA                                       │
│    - Запускаете локально, проверяете руками │
│    - Замеченные баги фиксите в той же       │
│      сессии (если контекст ещё в smart zone)│
│    - Или коммитите и идёте на новый цикл    │
├─────────────────────────────────────────────┤
│ 4. COMMIT & CLEAR                           │
│    - git commit                             │
│    - /clear                                 │
│    - Следующая задача — с чистого листа     │
└─────────────────────────────────────────────┘
```

### Подробнее по шагам

**Plan.** Стартуйте с короткого промпта (1–2 предложения). Если LLM сразу пытается выдать план без вопросов — прервите его (`Esc`) и попросите: *"interrogate me about every design decision before producing a plan"*. Качество плана пропорционально качеству диалога перед ним.

Когда план готов — почитайте его глазами на верхнем уровне. Если разговор был содержательным, обычно план хорошо отражает обсуждение, и детально вычитывать не обязательно.

**Решение о `/clear`.** После plan mode у вас может быть, скажем, 35% контекста. Решение:

- **Не чистить:** сохранить весь "дух" обсуждения. Риск — войти в dumb zone к концу имплементации.
- **Очистить:** агент работает только с планом. Риск — теряются нюансы, которые были в разговоре.

Эмпирическое правило: если остаётся < 60% контекста и задача мелкая → не чистить. Если задача большая → лучше очистить и стартовать с планом в качестве единственного контекста.

**Execute.** Включите `Shift+Tab` ещё раз чтобы агент мог редактировать без вашего подтверждения каждого шага (если вы доверяете). Следите за:

- Использует ли агент существующие модули или изобретает велосипед?
- Не появляются ли несуществующие импорты (галлюцинации библиотек)?
- Соблюдаются ли конвенции именования, размещения файлов?
- Прогоняет ли агент тесты и typecheck сам?

При замеченной проблеме — `Esc`, объясните, что не так, дайте докрутить.

**QA.** Никогда не пропускайте этот шаг. Запустите фичу руками, под разными ролями, на разных кейсах. Агент может иметь зелёные тесты, но фича может не работать так, как вы это представляли.

**Commit & Clear.** После QA — коммит, `/clear`, следующая задача. Дисциплина по очистке контекста — главное, что отличает зрелую работу с агентом от хаотичной.

## 9. Feedback loops: обратная связь как опора качества

> Великие инженеры не доверяют своим инстинктам и не доверяют качеству своего собственного кода. Они создают много feedback loops для повышения качества.

Та же логика применяется к агенту. Чем больше детерминированных проверок — тем выше качество вывода.

### Иерархия feedback loops

| Уровень | Скорость | Что ловит |
|---|---|---|
| Линтер / форматтер | мгновенно | стиль, очевидные ошибки синтаксиса |
| TypeScript / typecheck | секунды | несоответствие типов, отсутствующие импорты, опечатки в API |
| Юнит-тесты | секунды-минуты | логика отдельных функций |
| Интеграционные тесты | минуты | взаимодействие модулей |
| Запуск приложения локально | минуты | рантайм-баги, UX |
| Pre-commit hook | минуты | блокирует коммит на любом из вышеперечисленного |

Чем больше уровней работает у вас в проекте — тем меньше шанс, что агент выкатит что-то сломанное.

### `/do-work` skill

Этот skill оборачивает имплементацию в стандартный цикл. Скелет:

```markdown
---
name: do-work
description: Plan, implement, run feedback loops, and commit. Use for any non-trivial change.
---

When given a task:

1. **Plan** — sketch the change, identify modules touched.
2. **Implement** — write the code following existing patterns.
3. **Validate** — run `pnpm typecheck` and `pnpm test`. Fix any failures.
4. **Commit** — create a git commit with a clear message.

For backend code: use the Red-Green-Refactor pattern (see below).
For frontend code: implement directly without TDD; tests come later.
```

Когда вы говорите `/do-work` (или передаёте задачу со ссылкой на этот skill), агент следует чёткому ритму вместо того чтобы кидаться сразу писать код.

### Pre-commit hooks: детерминированный feedback

Husky + lint-staged даёт нам:

```
.husky/pre-commit:
  npx lint-staged
  pnpm typecheck
  pnpm test
```

`lint-staged` — пропускает Prettier (или ESLint --fix) только по staged-файлам, что быстро.

Что произойдёт, если агент попытается закоммитить сломанный код:

1. Hook отвергает коммит.
2. Stderr возвращается агенту: *"expected X to have a length of 999 but got 5"*.
3. Агент сам видит ошибку, фиксит, пробует снова.
4. Зелёный коммит проходит.

Для человека pre-commit hooks часто раздражают (никто не любит ждать 3 минуты). Для агента — идеально: ему всё равно, сколько времени ждать.

### Параметры тестирования, под которые удобнее жить агенту

- Тесты должны быть **изолированными**: не зависеть от внешних сервисов, не требовать поднимать всю инфру.
- Используйте in-memory базы (sqlite в режиме `:memory:`) или фикстуры в transactions с откатом.
- Имена тестов должны читаться как спецификация: `it("rejects expired coupons")` лучше, чем `it("test1")`.

## 10. TDD с агентом: Red-Green-Refactor

Test-driven development в исполнении агента работает удивительно хорошо. Цикл:

1. **Red.** Написать падающий тест. Запустить и убедиться, что он действительно падает (это важно — иначе тест ничего не проверяет).
2. **Green.** Написать минимальный код, чтобы тест прошёл.
3. **Refactor.** Привести код в красивый вид, не ломая тесты.
4. Повторить с следующим тестом.

### Почему это особенно работает с AI

- Агент **создаёт код, изначально testable**. Если тест нельзя написать — значит, код плохо спроектирован.
- Агент **может запустить код раньше, чем напишет имплементацию**: тест становится спецификацией поведения.
- Падающий тест → проходящий тест — это **видимая для вас траектория**. Агент трудно "симулирует" этот процесс — он либо реально работает, либо не работает.
- Множество тестов — это **сетка feedback loops**, которая ловит регрессии в будущих сессиях.

### Tracer bullet + TDD

Комбинация ещё мощнее: первый тест — это тонкий вертикальный срез всей фичи (один endpoint, один кейс). Не "спрей" тестов на всю поверхность, а **один тест за раз**.

```markdown
---
name: do-work
---

For backend code:
- Write ONE failing test for the smallest vertical slice
- Run it, see it fail (red)
- Write minimal implementation (green)
- Repeat with the next slice
- Refactor at the end if needed
```

В реальности агенты иногда пишут несколько тестов сразу, если уверены в реализации. Это не катастрофа, но если хочется строгой дисциплины — усиливайте инструкцию в skill'е.

### Когда TDD **не** применять

- **Frontend код.** Часто меняется в процессе UX-итераций; тесты на компоненты до их визуального утверждения — лишние.
- **Прототипы / спайки.** Цель прототипа — понять, что строить. Тесты приходят на этапе укрепления.
- **Интеграции с внешними API без хороших моков.** Можно начать с интеграционного теста и моком на верхнем уровне.

## 11. Архитектура кода под AI: deep modules

Идея взята из *A Philosophy of Software Design* Джона Аустерхаута. Ключевая мысль: модуль состоит из **интерфейса** (как его вызывают) и **реализации** (что внутри).

| Модуль | Интерфейс | Реализация |
|---|---|---|
| Shallow | сложный, много функций | мало кода за каждой функцией |
| Deep | простой, мало функций | много функциональности спрятано |

**Чем больше реализации спрятано за простым интерфейсом, тем лучше.**

### Почему deep modules важны для AI

1. **Читая интерфейс, агент сразу понимает функциональность модуля.** В shallow-кодбейзе агенту приходится прыгать между файлами, восстанавливая граф вызовов.
2. **Тесты живут на границе модуля**, а не на каждой внутренней функции. Они не ломаются от внутренних рефакторингов.
3. **Снижается когнитивная нагрузка на вас.** Когда AI постоянно меняет код, удерживать в голове детали реализации каждого мелкого модуля изматывает. Глубокие модули — "серые ящики": можете заглянуть внутрь, но обычно не приходится.

### Как идентифицировать кандидатов на углубление

- Несколько мелких сервисов, тесно связанных между собой и используемых вместе.
- Импорт-цепочки длинной более 3-4 шагов чтобы понять, что делает функция.
- Тесты, которые ломаются при любом изменении внутри модуля (а не только при изменении контракта).
- Модули, использующие разные паттерны для одного и того же (например, в одном месте raw SQL, в соседнем — ORM).

### Пример: квиз-подсистема

Было — три отдельных сервиса:
- `quizCRUDService` (создание, удаление квизов)
- `quizScoringService` (подсчёт баллов, использует raw SQL)
- `xpAwardingService` (начисление XP за прохождение)

Стало — один module с factory:

```ts
const quiz = createQuizModule({
  db,                    // injectable for tests
  scoringStrategy?,
  quizPassXp?,
});

quiz.getQuizForLesson(lessonId);
quiz.getBestAttempt(userId, quizId);
quiz.submitAttempt({ userId, quizId, answers });
```

Преимущества:
- Тесты инжектят in-memory базу, не нуждаются в моках.
- Все операции с квизами — за одним фасадом.
- XP-сумма становится параметром, а не зашита в код.

Этот рефакторинг был получен через `/improve-codebase-architecture` — skill сам предложил трёх вариантов интерфейса и аргументировал выбор.

### Module awareness в PRD

Хороший skill `/write-a-prd` теперь требует на стадии планирования:
1. Набросать модули, которые будут построены или изменены.
2. Активно искать возможности извлечь deep module, тестируемый изолированно.
3. Подтвердить с пользователем, что разбиение модулей соответствует ожиданиям.
4. Уточнить, для каких модулей и на какой границе писать тесты.

Это переносит архитектурные решения на этап планирования, не позволяя им "случиться" в процессе имплементации.

## 12. Кратко о Ralph (AFK-режим)

Поскольку вы пока не используете Ralph и не планируете на него опираться, привожу только концептуальный обзор — этого достаточно, чтобы понимать, о чём идёт речь в сообществе.

### Что такое Ralph

Ralph — это **простой bash-цикл** вокруг Claude Code. Не отдельный инструмент, а паттерн: вы запускаете агента с одним и тем же промптом снова и снова, пока задача не будет выполнена. Название — отсылка к Ральфу Виггему из "Симпсонов" (статья Geoffrey Huntley "Ralph Wiggum as a software engineer" сделала концепт известным).

### Зачем

Multi-phase plans требуют **человека в петле** — после каждой фазы вы говорите "do phase N". Это неэффективно, если задачи рутинные и не требуют ваших решений на каждом шаге. Ralph убирает человека из петли: вы запускаете цикл, идёте пить кофе, возвращаетесь к рабочему коду.

### Два варианта

| Скрипт | Поведение |
|---|---|
| `ralph/once.sh` | один запуск — режим human-in-the-loop, для обучения и наблюдения |
| `ralph/afk.sh` | цикл с лимитом итераций, полностью автономный |

Оба используют общий промпт-файл. AFK-версия обычно работает в Docker sandbox, чтобы агент был изолирован от вашей файловой системы (`docker sandbox run claude .`).

### Модель работы

Вместо передачи плана вручную — даёте Ralph backlog задач (например, GitHub Issues). Агент сам выбирает следующую задачу по приоритету, делает её, коммитит, переходит к следующей. Задачи помечаются как AFK (можно автономно) или HITL (требуется человек) — Ralph работает только с AFK.

### Когда стоит задумываться о Ralph

- У вас стабильный backlog с понятными мелкими задачами.
- Feedback loops хорошо настроены (тесты, типы, pre-commit) — иначе Ralph будет коммитить сломанное.
- Вы готовы инвестировать в инфраструктуру (Docker sandbox, GitHub CLI auth).

Если этого нет — оставайтесь на цикле Plan → Execute → Clear с человеком в петле. Большинству разработчиков этого хватает.

## 13. Практические советы и чеклисты

### Стартовый чеклист на новом проекте

- [ ] Установлен Claude Code и подключена подписка
- [ ] Установлен `ccstatusline` для отображения процента контекста
- [ ] Создан минимальный `CLAUDE.md` (не более 50 строк)
- [ ] Настроены `.claude/settings.local.json` с базовыми разрешениями
- [ ] Установлен `/grill-me`, `/write-a-prd`, `/prd-to-plan` (или `/prd-to-issues`), `/do-work`
- [ ] Pre-commit hook прогоняет typecheck и тесты
- [ ] Есть базовый тестовый раннер, проходящий локально

### Чеклист "вхождение в существующий проект"

- [ ] Запросил у Claude Code обзор: tech stack + назначение
- [ ] Изучил граф ролей и прав
- [ ] Понял, где живёт auth, где роутинг, где schema
- [ ] Запустил `/improve-codebase-architecture` — взял на заметку 1-2 кандидата на рефакторинг
- [ ] Описал в CLAUDE.md специфичные конвенции, которые отличаются от очевидных
- [ ] Проверил скорость typecheck и тестов; если медленно — это первый кандидат на улучшение

### Привычки на каждый день

1. **`/clear` между задачами.** Не подавляйте инстинкт.
2. **Контекст < 40%.** Если ушли выше — заканчивайте текущую задачу или коммитьте.
3. **Plan mode для всего нетривиального.** Даже 30-секундное обсуждение лучше, чем сразу execute.
4. **`@filename` вместо ручного копирования.** Быстрее и не съедает токены без необходимости.
5. **Не запускайте `claude init`.** Лучше написать CLAUDE.md руками с тем, что вам реально нужно.
6. **Каждое замеченное "грабли" — кандидат на skill.** Если та же ошибка повторяется в третий раз, оформите её как skill.
7. **Tracer bullets для всего нового.** End-to-end-срез на минимальных данных раньше, чем строить по слоям.

### Антипаттерны

- **Огромный CLAUDE.md.** Более 100 строк — почти наверняка перебор; разнесите по skills.
- **`/compact` как привычка.** Это техника редкого применения, не повседневная.
- **Игнорирование процента контекста.** "Ещё чуть-чуть и закончу" — классическая дорога в dumb zone.
- **Tests-after.** Если возможно, пишите тест **до** имплементации; агент справляется лучше.
- **Доверие к "зелёным тестам" без QA руками.** Тест может проверять не то.
- **Принятие плана с первого раза.** Хорошие планы рождаются в диалоге; задавайте вопросы, оспаривайте.
- **Пропуск explore-фазы в существующем проекте.** Новый код, не вписанный в существующие паттерны, — главный источник энтропии.

### Что почитать, если хочется глубже

- **A Philosophy of Software Design** — Джон Аустерхаут. Источник идеи deep modules.
- **The Pragmatic Programmer** — Хант, Томас. Источник tracer bullets и понятия software entropy.
- **Extreme Programming Explained** — Кент Бек. Про TDD и Red-Green-Refactor.

## Заключение

Главный сдвиг в работе с AI-агентами — **не тулинг, а инженерные привычки**. Тулы (Claude Code, skills, pre-commit hooks) — это амплификатор. Они усиливают то, что у вас уже есть: дисциплину писать тесты, аккуратно планировать, чисто декомпозировать модули. Если этого нет — амплификатор усилит и хаос.

Поэтому, если попытаться выжать суть в одно предложение: **ваш код должен быть таким, в котором AI приятно работать**. Это значит — хорошие имена, изолированные модули, быстрые feedback loops, документированные контракты. Всё это вы и так знали как "хорошие практики". Просто в эпоху агентов цена несоблюдения становится выше, а не ниже.

Удачной работы.
